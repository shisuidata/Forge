### Session 持久化

原来的 Session 是纯内存的——`dict` 存在进程里，进程一重启，所有用户的上下文全部丢失。用户问到一半的问题、还没执行的 SQL、待确认的指标定义，全部清零。这件事在开发阶段无所谓，但如果有人真的在用，重启一次等于强制对话中断。

改成 SQLite 持久化，写法是每次状态变更时触发回调：

```python
# session.py — SessionStore.get()
sess._on_change = lambda uid=user_id: self._persist(self._sessions.get(uid))

# Session 内部，任何字段修改后调用
def _touch(self):
    if self._on_change:
        self._on_change()
```

`_persist()` 用 `INSERT OR REPLACE` 写入 SQLite，序列化的是 `history` 和时间戳，`pending_sql` 等"进行中"状态不写入——这些状态如果进程崩溃就已经失效了，不值得恢复，强行恢复反而会产生歧义。

```sql
-- sessions 表
CREATE TABLE IF NOT EXISTS sessions (
    user_id   TEXT PRIMARY KEY,
    history   TEXT NOT NULL,
    updated_at REAL NOT NULL
)
```

测试的时候踩了一个坑：测试文件里复用了同一个全局 `store` 对象，不同测试写入同一个 `.forge/sessions.db`，状态相互污染，随机失败。修法是加了一个 `_tmp_store()` 辅助：

```python
def _tmp_store():
    return SessionStore(db_path=Path(tempfile.mktemp(suffix=".db")))
```

每个测试用自己的临时数据库。测试隔离是最基本的事，但在并发或有副作用的场景里，它比你想的更容易被忽略。

### 编译器拆分

`_coerce()` 是 Forge DSL 里负责"把合法但不完整的 JSON 补全规范形式"的函数。在某次重构之前，它是大约 390 行的线性流程——先处理顶层字段，然后 filter，然后 agg，然后 select，然后 group……每个部分都用注释分隔，但本质上是一段从头跑到尾的代码。

这本身不是 bug，但它让修改很危险：你改了第 150 行的 `agg` 处理，你不知道是否影响了第 280 行的 `having` 逻辑，因为中间可能有隐式的状态依赖。测试覆盖了输入输出，覆盖不了内部状态转移路径。

拆成 10 个独立子函数：

```python
def _coerce(q: dict, dialect: str) -> dict:
    q = _coerce_toplevel(q, dialect)
    q = _coerce_filters(q)
    q = _coerce_agg(q)
    q = _coerce_having_agg_refs(q)
    q = _coerce_select(q)
    q = _coerce_group(q)
    q = _coerce_cte_refs(q)
    q = _coerce_joins(q)
    q = _coerce_filter_vals(q)
    q = _coerce_window_qualify(q)
    return q
```

每个子函数只读它负责的字段，输入输出都是完整的 `q` dict，没有隐式状态。53 个测试全绿，3 个已知 pre-existing 的失败没有变化。

拆分不是为了增加功能，只是为了让未来的修改变得安全一点。这类工作很难看见价值，你做完之后系统表现完全一样，但你知道下次改的时候不会那么害怕。

### 飞书 Bot 的连锁故障

这是这一天最消耗精力的部分，不是因为复杂，而是因为错误是连锁的——修了第一层，才能看到第二层；修了第二层，才能看到第三层。

**第一层：embedding API 调用失败。**

Bot 在召回阶段崩溃，报错指向 `https://api.minimaxi.com/v1/embeddings`。问题是 `forge.yaml` 里的 `embedding.base_url` 残留了旧的 MiniMax 地址——之前换到硅基流动的时候只改了 `.env`，没有同步更新 yaml，而 `config.py` 的读取优先级是 `.env` > `forge.yaml` > 默认值，`.env` 里是对的，yaml 里是错的。但 yaml 在某个路径下被直接读取，没有走 `config.py` 的合并逻辑，所以 yaml 里的旧地址生效了。

找到原因之后感觉有点无语——配置错误，但不是"配置写错了"，而是"配置在两个地方，只改了一个"。这类问题在 debug 时特别难找，因为你会先假设环境变量是正确的，而它确实是正确的。

修完之后，第二层出现了。

**第二层：context window 超限。**

LLM 报错 `context window exceeds limit`。大 Schema 的 Registry JSON 有 200 张表，embedding 检索负责过滤出相关的 ~10 张表注入 prompt。但 embedding 检索失败了——`forge.yaml` 里的 `registry.*` 路径指向的是 `registry/data/`，那个目录在当前配置里是空的，Registry 实际上存在 `tests/datasets/large/`。

Retriever 初始化失败，静默降级——召回返回全部 200 张表，全量注入 LLM，撑爆 context window。

"降级"本来是容错机制，但这里降级的代价是把一个可预期的小错误（路径配置错误）放大成了一个不可预期的大错误（LLM 上下文超限）。这个设计值得商榷。

修完路径，第三层出现了。

**第三层：硅基流动 embedding 批量限制。**

`build_index()` 把 200 张表的所有文本一次性发给 SiliconFlow API，返回 422 报错——批量上限约 50 条。

修法是在 `build_index()` 里加分批循环：

```python
# forge/retriever.py
batch_size = 32   # 留余量，不踩边界
parts = []
for i in range(0, len(texts), batch_size):
    parts.append(embed_fn(texts[i:i + batch_size]))
raw = np.concatenate(parts, axis=0)
```

32 是留余量的安全值，不是精确的上限。

修完批量，第四层出现了。

**第四层：数据库 0 字节。**

查询执行成功，返回空结果集。把 SQL 贴到 SQLite CLI 里手动跑，报错 `no such table: orders`。查 `forge.yaml`，`database.url` 指向 `demo/large_demo.db`——这个文件存在，但大小是 0 字节。真正的数据库是 `tests/datasets/large/database.db`，9MB，134,577 行，200 张表。

改了 URL，同时删掉了所有 `.pkl` 缓存文件——旧的 embedding 缓存是 MiniMax 维度，和硅基流动不兼容，继续用会静默返回错误的向量。

四层修完，Bot 稳定了。

### demo-setup.sh 和 forge config

这是这一天唯一一件"从零开始设计"的事，其他都是修修补补。

`demo-setup.sh` 是一个交互式向导，用户 clone 之后运行，引导完成全部配置：

```
[1/5] 大模型配置     → Anthropic / OpenAI 兼容接口
[2/5] 向量模型配置   → 硅基流动 bge-m3 / OpenAI / 自定义 / 跳过
[3/5] 数据库连接     → Demo 200 表数仓 / 自有数据库
[4/5] 反馈机制       → 是否开启语义库自动维护
[5/5] 飞书 Bot       → 可选，填入 App ID 和 Secret
```

写到"数据库连接"这一步的时候意识到一个问题：选自有数据库的用户，向导结束之后会拿到一个完整的结构层（`forge sync` 自动生成），但语义层是空的——指标定义、歧义消除规则、字段使用约定，这些需要他们自己写，但他们不知道怎么写。

所以 `docs/registry.md` 这篇文档变得必要了。它不是补充说明，它是向导的一部分——选择自有数据库的用户会在向导结束时看到一行：

```
详见 docs/registry.md — 构建你自己的语义库
```

`forge config` CLI 是另一端的入口，给已经在用的用户提供配置修改能力，不用手动编辑 YAML：

```bash
forge config                          # 查看当前所有配置（API Key 打码）
forge config llm.model                # 查看单个 key
forge config llm.model deepseek-chat  # 写入 forge.yaml
```

配置项通过点分路径写入，`database.url` 修改后会额外提示运行 `forge sync`。

## 往深处想

### 配置是产品的第一印象

连锁故障修完之后，我想了一下这件事为什么会发生。

本质上是两套配置系统在同一个地方生效——`.env` 是环境变量，`forge.yaml` 是应用配置，`config.py` 负责合并，优先级是 `.env` > yaml > 默认值。这个设计是对的，它允许用户把 API Key 放 `.env`（不进 git），把其他配置放 yaml（可以进 git）。

问题在于，我在多个地方直接读 YAML，没有走 `config.py`。这打破了唯一真实来源的原则，配置开始在多个地方各自生效，最终变成了"改了 A，B 还是老样子"。

工具理解它自己的配置，这件事听起来很基础，但如果配置有多个来源、多个读取路径，每增加一个路径就增加了一个潜在的不一致点。`forge config` 的价值不只是提供一个好用的 CLI，更重要的是它强制所有配置写入通过同一个路径——`forge.yaml`，由 `config.py` 统一读取。

这个教训可以更简洁地表达：如果用户能在两个地方配置同一件事，他们最终一定会在一个地方改、另一个地方忘记改。

### 门面不是粉饰

Phase 3 的工作大部分是文档和脚本，没有改动核心逻辑。在开始做之前，我有一个模糊的不适感：这算不算"真正的工程工作"？

做完之后，这个不适感消失了。

`demo-setup.sh` 的第三步——用户选择连接自有数据库——迫使我想清楚两件事：第一，用户同步了结构层之后会立即遇到什么困难（语义层是空的，不知道怎么填）；第二，我们有没有给他们一条出路（`docs/registry.md` 的那个链接）。

没有这个向导，这两件事我可能想到，也可能想不到。向导把用户路径具体化了，迫使你在写代码的时候同时扮演用户，而不是在写完代码之后再补文档。

README 也是同一件事。旧的 README 有 600 多行，把所有东西堆在一起。改成 170 行之后，我被迫思考：对于第一次看到 Forge 的人，哪一段话最重要？什么应该在首屏，什么应该在链接里。这个决策本质上是产品决策，不是写作决策。

门面不是粉饰，它是你对用户路径的判断。

### 连锁故障的形状

四层连锁故障有一个共同的结构：每一层都有一个"表面上正常"的行为屏蔽了真正的错误。

- embedding API 失败 → 表面上看是 MiniMax 地址问题，实际是 yaml 和 env 不同步
- Retriever 初始化失败 → 表面上看是召回变慢，实际是全量表注入撑爆了 context
- 批量 API 报错 → 表面上看是 embedding 失败，实际是批量限制没有文档
- 数据库 0 字节 → 表面上看是查询返回空，实际是数据库文件不存在

每一层的"表面正常"都延迟了对真实错误的定位。如果 embedding 失败时直接报错退出，而不是降级为全量召回，第二层的 context 超限就不会发生，我可以在第一步就定位到真正的问题。

这让我对"优雅降级"这个设计原则有了一些新想法。降级是正确的——在 embedding API 不可用时，用全量表作为兜底比直接崩溃要好。但降级不等于静默。正确的做法可能是：降级 + 明确警告，让用户知道系统现在在降级模式运行，而不是假装一切正常。

现在的代码里，降级时会打 `logger.warning`，但飞书 Bot 界面上用户看不到。下一步可能需要在降级时给用户发一张提示卡片——不是报错，只是告诉他们"当前使用的是完整 Schema，召回精度较低"。

### 向导作为规格文档

写 `demo-setup.sh` 的过程里，我发现自己在写一份配置规格文档——不是面向开发者的规格，而是面向用户的规格。

第二步的向量模型配置里，我写了这段话：

```
大表场景下（>10 张表），Forge 用向量检索只注入相关表到 prompt，
避免 context window 溢出。小表场景可跳过。
```

这句话里有两个隐含的决策：什么算"大表场景"（>10 张），embedding 对小表场景的性价比（不值得）。这些决策在代码里是硬编码的，但之前从来没有被明确写出来过。写向导的时候，我被迫把它们变成了可以向用户解释的语句。

这个过程有点像是把代码里的隐式假设显式化。代码里有很多这类假设——哪个阈值是经验值、哪个默认值是历史遗留、哪个边界条件是 corner case 还是主路径。这些假设在写代码时是清楚的，但时间一长，连自己也会忘记。

向导迫使我用用户语言重新表达了一遍这些假设，结果是：我发现了几处"我以为我知道，但其实说不清楚"的地方。

## 悬而未决

**Qwen2.5-7B 的基准测试还没跑。** `method_p.py` 和 `method_b_qwen.py` 都写好了，但需要 `SILICONFLOW_API_KEY` 环境变量，还没有设好跑完整的 40 题。我想知道的是：同等参数量的模型，Forge 约束到底能把准确率提多少。DeepSeek V3 在大 Schema 上是 65%，小模型会是多少——是 30% 还是 50%，差距有多大。

**降级时应该对用户可见。** 现在所有的降级路径（embedding 失败回全量、Retriever 初始化失败回全量）只打 warning 日志，用户看不到。这在开发阶段可以接受，但在真实使用场景里，用户应该知道 Forge 现在的召回质量是不确定的。

**Session 持久化改变了对话的基本约定。** 现在重启进程之后，用户的历史记录会被恢复——上次问了什么、还在等待执行的 SQL 确认。但"等待执行"的状态在重启后不可恢复，只有历史消息会恢复。如果用户重启前发了"查一下上个月的订单量"，重启后的历史里会有这条消息，但 `pending_sql` 是空的——继续对话会出现不一致。这个边界目前没有处理。

**大 Schema 下的 65% 是什么在失败。** 这个数字还停在上次跑完基准的结果里，没有做更细致的分析。失败的那 35%，是召回选错了表，是编译器生成了错误的 SQL，还是 LLM 根本没有理解查询意图？三种失败的修法完全不同。我一直想坐下来好好分析，但一直有其他事情插进来。

## 这一天的感觉

以前觉得，写文档和向导是"开发之外的事"——真正的工作是写代码，文档是之后补的。

这一天改变了这个感觉，但改变的方式不是我预期的。我预期的是"文档很重要，要重视"这样的顿悟。实际发生的是：我在写向导第三步的时候，发现需要加一个"语义库为空时给用户的提示"，然后去写 `docs/registry.md`，写的时候发现原来的文档有一些说明不准确，修的时候发现代码里对应的逻辑也需要调整。

文档和代码是同一件事的两面。把文档推到最后，就是把一部分设计决策推到最后——推到用户已经拿到产品、你才发现问题的那一刻。

这不是结论，只是今天学到的东西。
