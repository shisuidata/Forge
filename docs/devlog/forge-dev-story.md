# Forge 开发实录【day 0】：一个数据人的 AI 查询 Agent 建造之旅

> 写于 2026 年 3 月。这不是一篇成功学文章，也不是产品发布会演讲稿。这是一份真实的开发记录，包含所有走错的弯路、自我怀疑的深夜、以及偶尔出现的顿悟时刻。

---

## 一、引言：为什么要做这件事

我在数据行业工作了很多年。这些年里，我见过太多种类的工具被发明出来，宣称要"解放数据分析师"——BI 工具、可视化平台、低代码分析引擎，每一个都带着"让业务人员自助分析"的承诺登场，然后悄悄地消失在企业内网的某个角落。

LLM 的到来让这个赛道重新热闹起来。从 2023 年开始，"Text-to-SQL"成了 AI 应用最高频出现的词汇之一。演示视频里，你只需要用自然语言问一句"上个月各城市的销售额是多少"，AI 就会漂亮地输出一段 SQL，数据就出来了。看起来，这一次真的要成了。

然后，真实的数据团队开始用这些工具。

我见过太多次这样的场景：一个数据分析师兴冲冲地把 Text-to-SQL 工具接进公司的数据仓库，试了几个查询，觉得不错，开始推广。然后业务部门的人开始用，某一天他们拿着一份数据报告来找数据团队核实，说这个数字好像不对。数据团队一查，发现 AI 生成的 SQL 里，一个应该用 LEFT JOIN 的地方写成了 INNER JOIN，直接把没有订单记录的用户全部过滤掉了。那份报告里的用户留存率，因此少了二十个百分点。

更糟糕的是，这类错误不会报错。SQL 执行成功，数据也出来了，只是数字是错的——安安静静地错着，直到有人发现。

这就是我做 Forge 的起点：一种对于"静默错误"的愤怒。

不过在写代码之前，我先做了一件对后续影响最深的事：把问题想清楚。

### 问题的本质

为什么 LLM 会在 Text-to-SQL 任务上出错？这个问题比看起来复杂。我花了两周时间整理自己见过的所有 SQL 生成失败案例，试图找到规律。

最终，我把错误分成了三类，这个分类在整个 Forge 开发过程中作为北极星一直没有变过：

**第一类：生成错误。** 模型的推理是对的，它知道用户要什么，但是把这个意图翻译成 SQL 的时候出了岔子。典型的例子：
- 用 `INNER JOIN` 替代 `LEFT JOIN`，把应该保留的空值行过滤掉了
- `NOT IN` 子查询里有 NULL 值，导致整个结果集静默为空
- 忘了写窗口函数的 `OVER` 子句
- `WHERE` 里写了聚合条件（`WHERE COUNT(*) > 5`），应该是 `HAVING`
- JOIN 没有写类型，裸 `JOIN` 等价于 `INNER JOIN`，但意图可能是 `LEFT JOIN`

**第二类：业务逻辑错误。** 模型的推理也没问题，但它无法知道你们公司对某个指标的特定定义。比如"复购率"——在某些公司，分母是所有注册用户；在另一些公司，分母是下过至少一单的用户。这两种定义算出来的数字差距巨大。模型在没有上下文的情况下，只能猜。猜中了是运气，猜错了就是数据事故。

**第三类：算法逻辑错误。** 这个最难，也是我一开始就决定放弃解决的一类。比如"生成过去 30 天每天的日期序列，包括没有数据的日期"——这需要用 `generate_series` 或者递归 CTE 来造一张日历表，然后和业务数据 LEFT JOIN。这不是一个翻译问题，这是一个算法问题，需要了解具体的实现方法才能作答。模型可能知道，也可能不知道，这取决于它的训练数据，而不是约束和提示。

这个分类有一个重要的推论：对于第一类和第二类错误，我们可以系统性地消灭它们；对于第三类错误，我们应该诚实地承认边界，而不是假装可以解决。

这听起来简单，但在 AI 应用的语境里，"承认自己解决不了某类问题"是需要勇气的。整个行业的默认姿态是"更好的 prompt 可以解决一切"，或者"下一个版本的模型会更强"。我在 Forge 里选择了另一条路：把能解决的问题解决好，把不能解决的问题清楚地标注出来。

### 核心洞见的形成

问题分类之后，我开始思考解法。真正让我找到方向的，是一个听起来有点绕的观察：

**LLM 的错误率与输出空间大小正相关。**

当你让 LLM 直接写 SQL，它面对的是一个几乎无限大的输出空间——任意的表名、任意的列名、任意的 SQL 语法组合。在这个巨大的空间里，正确的 SQL 只是一个极小的点，而错误的 SQL 几乎无处不在。

如果我们能大幅收窄输出空间呢？

如果我们能让 LLM 只从 Registry 里存在的表名和列名中选择——那幻觉列名就物理上不可能出现。

如果我们强制 JOIN 类型必须从一个枚举值里选——那裸 JOIN 就物理上不可能出现。

如果我们把 WHERE 条件和 HAVING 条件分成两个不同的字段——那混淆就物理上不可能出现。

如果我们用 `anti` 这个 JOIN 类型来代替 `NOT IN`，由编译器保证它被翻译成正确的 `LEFT JOIN ... WHERE right_key IS NULL`——那 NULL 陷阱就物理上不可能出现。

这就是 Forge 的核心思路：**不是让 LLM 写出更好的 SQL，而是设计一个中间语言，让某些错误在物理上不可能发生。**

这个思路后来让我找到了学术先例：2019 年 ACL 会议上，IRNet 论文提出了 SemQL（一种树状的语义表示语言），用"中间语义表示 + 确定性编译"的路径取代直接生成 SQL，在 Spider benchmark 上比直接生成 SQL 提升了 19.5 个百分点。Forge DSL 是同一理念的工程化实现，只不过更现代，使用 JSON 格式，用 JSON Schema + Structured Output 来施加约束，而不是特殊 token 文法。

在读到这篇论文的那个下午，我有一种奇特的感觉：不是"啊，有人已经做过了"的沮丧，而是"我的直觉是对的"的确认。

但 IRNet 和 Forge 之间仍然有很大的差距，不是思路上的，而是工程化程度上的。IRNet 是一篇学术论文，它的目标是在特定 benchmark 上展示 SOTA 效果，而不是构建一个可以在企业内部署的系统。学术 benchmark 的 schema 都是预先定义好的，不存在 Registry 的问题；学术 benchmark 不需要跨方言支持；学术 benchmark 不需要容错机制（编译失败就是失败，没有 `_coerce` 修复）。

Forge 要解决的是在真实企业场景里让这套机制实际可用：不同的方言、用户自己的数据库 schema、模型输出的格式偏差、复杂的业务语义……这些"工程化的脏活"，是 IRNet 这类学术工作里完全没有涉及的部分。

认识到这个差距，让我对接下来的工作有了更清晰的期待：我在做的不是理论创新，而是工程化落地。这件事本身的价值不亚于理论创新，但衡量标准不同——不是"有没有提出新的方法论"，而是"在真实场景里能不能真正减少错误"。

## 二、问题定义：SQL 生成错误的本质

在开始写代码之前，我做了一件对项目影响最深远的事：写了一套针对性的失败案例测试集。

这些测试用例不是从学术 benchmark 上抄来的，而是从我自己在项目里见过的真实错误中整理出来的。我在 `tests/text-to-sql-failures/` 目录下建了一套场景化的案例，每个案例对应一种具体的错误模式。

测试 schema 故意做得很简单：

```sql
users        (id, name, city, created_at, is_vip)
orders       (id, user_id, status, total_amount, created_at)
order_items  (id, order_id, product_id, quantity, unit_price)
products     (id, name, category, cost_price)
```

四张表，二三十个字段，覆盖了 E-commerce 场景下绝大多数常见的分析查询。这个 schema 的选择是刻意的：如果在这么简单的 schema 上都频繁出错，说明问题在于生成机制本身，而不是 schema 的复杂度。

案例覆盖的错误类型包括：

| 案例 | 错误类型 | Forge 的解答策略 |
|---|---|---|
| A1：LEFT JOIN vs INNER JOIN | 生成错误 | DSL 强制声明 JOIN 类型 |
| A2：NOT IN + NULL 陷阱 | 生成错误 | `anti` join 原语，编译为 LEFT JOIN + IS NULL |
| B1：WHERE vs HAVING 混淆 | 生成错误 | `filter` 和 `having` 分离为独立字段 |
| B2：与自身均值比较 | 算法逻辑错误 | 诚实承认超出能力边界 |
| D1：复购率指标歧义 | 业务逻辑错误 | Registry 语义层 |
| E1：每组取 TopN | 算法逻辑错误 | 诚实承认超出能力边界 |
| E2：窗口函数语法错误 | 生成错误 | DSL 将窗口函数结构化 |

这套案例集在整个开发过程中是我的地基。每次改动 prompt 或者编译器，我第一件事就是跑这些测试，看有没有退步。

### 为什么选择这个 Schema

在确定测试 schema 的时候，我考虑过用更复杂的场景，比如金融或者医疗数据。但最终选择了这套 E-commerce schema，理由很简单：

1. 人人都能理解，不需要领域知识就能判断 SQL 是否正确
2. 覆盖了绝大多数常见的关系代数操作
3. 枚举值足够有趣（`status` 有 `completed` 和 `cancelled`，`is_vip` 有 0 和 1），可以测试 Registry 的枚举约束效果

这个选择在后来的 Spider2-Lite 测试中被证明是对的——学术 benchmark 往往使用更复杂的 schema，但那些复杂性中有很大一部分来自算法难度，而不是关系模型的复杂度。Forge 应该在简单 schema 上做到近乎完美，而不是在复杂 schema 上只能差强人意。

还有一个更实用的原因：这个 schema 足够小，我可以把每道测试题的 ground truth SQL 手写一遍，仔细验证正确性。如果 schema 太复杂，写 ground truth SQL 本身就会变成一项容易出错的工作，这会污染评测结果。

测试题的另一个设计原则是：每道题只覆盖一种主要的错误类型。在测试集设计里，最常见的错误是"一道题覆盖太多特性"——这会让你在分析失败原因时无法确定是哪个特性导致了问题。比如"找出各城市 VIP 用户的最近一次订单，按金额降序，只看已完成订单"这道题，它同时测试了 JOIN、过滤、窗口函数、排序。如果这道题失败了，你不知道是哪个部分导致的。

Forge 的测试题尽量做到"一道题，一个主要考点"，这让失败分析变得容易得多：当某道题失败，目标是明确的，修复方向是清晰的。

### SQL 的设计缺陷作为机会

在整理错误案例的过程中，我逐渐意识到：很多"生成错误"其实根植于 SQL 本身的设计缺陷。这些缺陷不是 bug，而是 1970 年代设计决策留下的历史包袱。

**书写顺序 ≠ 执行顺序。** SQL 的书写顺序是：

```sql
SELECT   -- 写在最前
FROM     -- 数据来源
WHERE    -- 过滤
GROUP BY -- 分组
HAVING   -- 聚合过滤
ORDER BY -- 排序
LIMIT    -- 限制行数
```

但 SQL 的实际执行顺序恰好相反：

```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

这意味着，当你在写 SQL 的时候，你实际上是在倒着思考执行流程。这是 SQL 最大的认知税之一。Donald Chamberlin 在 1974 年设计 SEQUEL（SQL 的前身）时，有意把 SELECT 放在最前面，是为了让 SQL 读起来更像自然语言（"给我选出...从...这张表...满足...条件"）。这个决策让 SQL 对初学者更友好，但也制造了大量的混淆，比如经典的"我能不能在 WHERE 里用 SELECT 里定义的 alias"——答案是不能，因为 WHERE 比 SELECT 先执行。

**NULL 三值逻辑。** SQL 的 NULL 不是 `false`，也不是空字符串，它是一个表示"未知"的特殊值。`NULL = NULL` 返回的不是 `true`，而是 `NULL`。`NULL != NULL` 也是 `NULL`，不是 `true`。这导致了 `NOT IN` 的经典陷阱：

```sql
SELECT * FROM orders WHERE user_id NOT IN (SELECT id FROM banned_users);
-- 如果 banned_users 里有一行 id 是 NULL，结果集静默为空
```

这是因为 `NOT IN` 等价于 `user_id != v1 AND user_id != v2 AND ...`，当子查询包含 NULL 时，`user_id != NULL` 的结果是 NULL（不是 true），所以整个条件变成了 NULL，没有任何行通过过滤。这个陷阱几乎每个用 SQL 写后端的工程师都踩过至少一次。

**GROUP BY 跨方言行为不一致。** 在 MySQL 的旧版本里，`SELECT * FROM t GROUP BY city` 是合法的，每组返回哪一行是不确定的（取决于数据库引擎的内部实现）。但同样的 SQL 在 PostgreSQL 里会直接报错：`column must appear in the GROUP BY clause or be used in an aggregate function`。这导致了大量在本地跑通、在生产环境报错的 SQL。

这些缺陷恰好对应了 Forge DSL 的设计机会：

- 书写顺序 = 执行顺序：Forge DSL 的字段顺序严格对齐 SQL 执行顺序（`scan → joins → filter → group → agg → having → window → qualify → select → sort → limit`），消除认知摩擦
- NOT IN 的 NULL 陷阱：用 `anti` join 原语代替，由编译器保证正确翻译
- GROUP BY 不一致：`_coerce` Fix 5 在编译前自动将 SELECT 中的非聚合列补齐到 GROUP BY

当我把这些设计决策写进 DSL 形式化语义文档的时候，我意识到：Forge DSL 其实是在修补 SQL 五十年来留下的一些坑。不是全部的坑，但至少是那些在 LLM 生成场景下最频繁被踩的坑。

## 三、核心洞见：约束论

在写第一行代码之前，我需要回答一个更基本的问题：为什么约束会有帮助？

这个问题看起来简单，但它有一个不那么显然的答案。

### 错误率与输出空间的关系

直觉上，我们总觉得"越灵活的工具越强大"。一个能写任意 SQL 的 LLM，应该比一个只能在固定格式内生成代码的 LLM 更厉害。但是，错误率是另一回事。

想象两个弓箭手，一个用标准的弓箭，靶心是个小圆圈；另一个用一把没有瞄准器的弓，可以射向任意方向。第二个弓箭手的"自由度"更高，但他命中靶心的概率也更低——因为"正确"的方向只有一个，而"错误"的方向有无数个。

LLM 生成 SQL 也是一样的道理。当输出空间是"任意 SQL"的时候，正确的 SQL 只有一个（或者少数几个等价的），而错误的 SQL 无穷多。当输出空间被收窄到"合法的 Forge JSON"时，约束已经物理上排除了大量错误可能性。

这不是说 LLM 随机游走——实际上，好的 LLM 在 SQL 生成任务上远比随机要好。但它确实会犯特定类型的错误，这些错误呈现出可预测的模式。Forge 的约束针对的就是这些已知的错误模式。

### 意图与执行的分离

约束的另一个好处是实现了"意图层"和"执行层"的分离。

在 Forge 的架构里：
- **LLM 负责**：理解用户的自然语言意图，把它翻译成 Forge JSON（语义层）
- **编译器负责**：把 Forge JSON 确定性地翻译成 SQL（执行层）

这个分离有几个深远的好处：

**可审计性。** 用户最终看到并审核的 SQL，就是实际会被执行的那段 SQL。没有任何"运行时魔法"，没有任何后台的二次处理。这在企业数据场景里非常重要——数据团队不仅需要结果正确，他们还需要能够向业务方解释"这个数字是怎么算出来的"。

**可调试性。** 当 SQL 输出了错误的结果，必然是 Forge JSON 的某一部分导致的，可以精确定位。这比在一大段黑盒 prompt 里找问题容易得多。

**可升级性。** 换一个更强的 LLM，不需要动编译器；优化编译器的某个 `_coerce` 修复，不需要重新训练模型。这两个组件是解耦的。

**可测试性。** 编译器是纯函数，相同的输入永远产生相同的输出。这意味着可以用单元测试充分覆盖它。当编译器出现 bug，可以写一个确定性的测试用例来重现并验证修复。而 prompt 的效果取决于 LLM 的随机性，很难用单元测试覆盖。

### 正向飞轮：Registry

约束论的第三个支柱是 Registry——组织知识的沉淀。

一个典型的企业数据环境里，存在大量"只有内部人才知道"的知识：
- `status` 字段的值是 `completed` 还是 `done`？
- "复购率"的分母包不包括从未下单的用户？
- "活跃用户"是指 30 天内有登录，还是 30 天内有购买？
- `is_vip` 是 boolean 还是 int？1 表示 VIP 还是 0 表示 VIP？

这些知识如果不被显式记录，LLM 只能猜。Registry 是让这些隐性知识变成显性知识的机制：

- **结构层**：`forge sync` 直连数据库，自动采样低基数列的枚举值。`status[cancelled/completed]` 这样的注解直接告诉 LLM 正确的字符串拼写。
- **语义层**：通过对话维护业务指标的精确定义，用一次准确一次，知识沉淀在系统里而不是在员工的脑子里。

Registry 的关键特性是**正向飞轮**：每次用户修正 Forge 的错误（比如"复购率的分母应该是有过购买记录的用户，不是所有用户"），这个知识就被写入 Registry，下次同样的查询就不会再出错。用得越多，Registry 越准确，错误率越低。

这和很多 AI 工具的体验相反——大多数 AI 工具每次对话都从零开始，永远不会从历史错误中学习。Forge 的 Registry 设计是故意要打破这个局面的。

## 四、DSL 的诞生：从 Version A 到 Version D 的大跳跃

所有的设计哲学都想清楚了，现在得写代码了。

第一个问题：DSL 用什么格式？

### Version A：SQL 风格 DSL（基线）

我最初的想法，是设计一种"简化版 SQL"——保留 SQL 的关键概念，但去掉那些容易出错的部分。大概长这个样子：

```
query {
  from: orders
  join: users on orders.user_id = users.id type=left
  filter: orders.status = "completed"
  group: users.city
  agg: count(*) as order_count, avg(orders.total_amount) as avg_amount
  select: users.city, order_count, avg_amount
  sort: avg_amount desc
  limit: 10
}
```

这是 Version A，一种类似 GraphQL 风格的查询语言。我花了一周时间写了解析器和编译器，然后跑了一轮评测。

结果：**LLM 评分 7.63，编译失败率 3.8%。**

7.63 分（满分 10 分）对应的是什么水平？大概是"大多数情况下能用，但有明显瑕疵"。3.8% 的编译失败率意味着每 26 次查询里有一次直接编译失败，连 SQL 都出不来。

更糟糕的是：作为对照组，我让同一个 LLM 直接生成 SQL（不经过任何中间层），得分是 **8.38**。直接生成 SQL 比我的 DSL 还要好。

这是个令人沮丧的数字。我花了将近两周设计和实现这套 DSL，结果反而不如没有它。

但是，在仔细分析失败案例之后，我找到了问题的根源：**格式本身就是问题。**

自定义的文本 DSL 有几个致命弱点：
1. LLM 不熟悉这种格式——它在训练数据里见过大量 SQL，但几乎没有见过我发明的这套语法
2. 文本 DSL 的约束是"软性的"——即使我在 prompt 里写了"join 必须声明 type"，模型也可以无视这条规则，因为没有任何机制强制它
3. 解析文本 DSL 需要写解析器，而解析器本身就是错误的来源（编译失败率 3.8% 很大程度上来自解析阶段）

### 关键转折：从文本 DSL 到 JSON DSL + Structured Output

版本 B 和 C 我跳过不讲（它们是各种变体的尝试，都没有实质性改进），直接说 Version D：

**用 JSON DSL + Structured Output（token 级别约束）。**

这个想法来自于思考 Structured Output 的本质。Structured Output（OpenAI 的叫法，其他 API 厂商也支持类似功能）是一种让 LLM 输出严格符合 JSON Schema 的 JSON 的机制——不是"尽量符合"，而是在 token 生成层面强制约束，生成的每个 token 都必须使 JSON 保持合法。

这意味着：
- 如果 JSON Schema 定义 `fn` 字段是枚举值 `["count", "count_all", "sum", "avg", "min", "max"]`，那么生成的 JSON 里 `fn` 字段就**物理上不可能**出现 `"total"` 或者 `"COUNT_STAR"` 这样的幻觉值
- 如果 JSON Schema 定义 JOIN 的 `type` 是枚举值 `["inner", "left", "right", "full", "anti", "semi"]`，那么裸 JOIN（无类型）**物理上不可能**出现

这和文本 DSL 的"软性约束"是本质不同的。

决定转向 JSON DSL 之后，我重新设计了整个格式。核心原则：

1. **执行顺序 = 字段顺序**：`scan → joins → filter → group → agg → having → window → qualify → select → sort → limit`，与 SQL 实际执行顺序一致
2. **所有枚举值用 JSON Schema 强制**：JOIN 类型、聚合函数名、排序方向、窗口函数类型，全部是枚举
3. **语义分离**：`filter`（WHERE）和 `having`（HAVING）是两个独立字段，不可能混淆
4. **anti join 代替 NOT IN**：`type: "anti"` 由编译器翻译为 `LEFT JOIN ... WHERE right_key IS NULL`

Version D 的评测结果：**LLM 评分 8.46，编译失败率 1.2%。**

相比 Version A：**+0.83 分，编译失败率从 3.8% 降到 1.2%。**

这是整个项目中单次最大的跳跃。

0.83 分看起来不多，但在 LLM 评分的量纲里，这意味着几乎每道题都有提升。更重要的是，它让 Forge 的得分（8.46）超过了直接 SQL 生成（8.38）——这是 Forge 第一次证明自己比"不用 Forge"要好。

这个跳跃背后的本质是：**从"希望 LLM 遵守规则"到"物理上不可能违反规则"。**

### JSON Schema 的设计哲学

`schema.json` 是 Forge 最重要的设计决策的集中体现。这个文件有 500 多行，定义了 Forge DSL 的完整结构。

几个重要的设计选择：

**`additionalProperties: false`（到处都用）。** 这意味着 JSON 里任何不在 schema 定义里的字段都会被拒绝。这看起来很死板，但它防止了模型"发明字段"。模型有时候会在 agg 项里加上 `"where"` 或者 `"having"` 字段，这不是合法的 Forge JSON，`additionalProperties: false` 直接把这类情况挡掉了。

**`count_all` 和 `count` 分开定义。** `count_all` 编译为 `COUNT(*)`，没有 `col` 字段；`count` 编译为 `COUNT(col)`，必须有 `col` 字段。这消灭了 `COUNT(*)` 和 `COUNT(col)` 的混淆（两者在含 NULL 值的列上结果不同）。

**JOIN 类型枚举包含 `anti` 和 `semi`。** 这两种不是 SQL 里的关键字，是 Forge DSL 里的原语。`anti` 编译为 `LEFT JOIN ... WHERE right_key IS NULL`，`semi` 编译为 `WHERE EXISTS (SELECT 1 FROM ...)`。通过给这两种高风险的操作起名字，让 LLM 可以明确表达这两种意图，而不是在可能出错的 `NOT IN` 和各种变体之间猜。

**`explain` 字段。** 这是一个不参与编译的字段，模型可以在里面写一段自然语言解释它的意图。这个字段有两个用处：一是帮助调试（当编译失败时，`explain` 字段提供了理解 LLM 意图的上下文）；二是在错误恢复时，把 `explain` 连同错误信息一起回传给 LLM，帮助它自我修正。

## 五、Prompt 炼狱：版本 E 到 J 的踩坑记

从 Version D 开始，DSL 格式基本稳定，进入了漫长的 prompt 迭代阶段。

我在这一阶段的主要工具是一套自动评测流水线：把 40 道测试题跑 5 遍，用 LLM 作为 judge 打分（0-10 分），取平均值。每一轮迭代，我都记录了分数的变化、编译失败率，以及具体哪些题目有提升或退步。

这段经历很折磨，因为 prompt 工程充满了蝴蝶效应：修好了一个案例，经常会同时破坏另外两个。

### Version E：第一次退步（-0.05）

Version E 的目标是处理 HAVING alias、LIMIT、排名这几类问题。我认为这些案例在 D 上表现不佳，于是加了一些针对性的 prompt 指导。

结果：**8.46 → 8.41，-0.05。**

退步了。

分析原因：HAVING alias 的补丁破坏了部分正常工作的 GROUP BY 案例。我加的关于 HAVING 的规则太宽泛，影响了模型在其他情况下的判断。

这是 prompt 工程的第一课：**加新规则之前，要想清楚它在所有情况下的副作用，而不只是它解决的那个问题。**

### Version F：语义精确（+0.02）

Version F 尝试修正 semi join 的语义（从 IN 子查询改为 EXISTS 模式）和 JOIN 完整性（所有 JOIN 字段必须显式声明）。

结果：**8.41 → 8.43，+0.02。**

几乎没有提升，但至少没退步。

这轮迭代里我踩了第二个坑：我加了一条负向规则，大意是"不要在 HAVING 里幻造不存在的聚合条件"。

这听起来是正确的，因为我见过模型在 HAVING 里编造不存在的聚合函数（比如"当平均金额大于 800 元时过滤"，但 agg 里根本没有定义这个平均金额的聚合）。

但结果是：Case 4 和 Case 9 都退步了。这两道题本来是正常工作的，在加了这条负向规则之后，模型变得过于保守，不敢写 HAVING 条件，即使 HAVING 是正确的选择。

**负向规则（"不要做 Y"）的危险性：它不仅阻止了错误行为，还会误伤正确行为。**

### Version G：正向规则的力量（+0.26，最大单次 Prompt 提升）

这一轮我做的最重要的改变，是把负向规则全部替换成正向规则。

比如，之前写的：
> 不要在 HAVING 里幻造不存在的聚合条件

替换成：
> HAVING 的 col 字段必须引用 agg 里定义的某个 alias；如果需要的聚合值不存在于 agg 中，先在 agg 里添加它

之前写的：
> 不要把 WHERE 条件放在 HAVING 里

替换成：
> filter 字段对应行级过滤（WHERE），有聚合时用 having 字段；filter 在 GROUP BY 之前执行，having 在聚合之后执行

这一轮还处理了数量词语义的问题。"超过 5 次"应该是 `gt`（严格大于），而不是 `gte`（大于等于）。这个细节在 prompt 里很难用负向规则处理，但用正向规则就清晰得多："'超过 N 次'对应 `op: gt`；'至少 N 次'对应 `op: gte`；'不超过 N 次'对应 `op: lte`"。

结果：**8.43 → 8.69，+0.26。**

这是整个 prompt 迭代阶段最大的单次提升。

**工程洞察：正向规则（说"正确做法是 X"）比负向规则（说"不要做 Y"）更稳定。** 正向规则告诉模型往哪里走，负向规则只告诉模型哪里不能去——但模型会倾向于把"不能去的地方"解读得比你预期的更大，从而误伤邻近的正确行为。

### Version H：新能力导致过拟合（-0.24）

Version G 之后，我开始考虑扩展 DSL 的能力。我加了三个重要的新特性：
- `CASE WHEN` 在 agg 内部的支持（用于条件聚合）
- `$preset` 相对日期（"上个月"、"本季度"这类时间筛选）
- CTE（公共表表达式，用于多步骤聚合）

这些功能本身是有价值的。但是，我在 prompt 里加了完整的 CTE 使用文档之后，出现了一个意想不到的问题：

**模型开始把简单的 GROUP BY 查询也包装成 CTE。**

比如，"统计各城市的订单数量，按数量降序排列"这样的查询，原本一个简单的 `GROUP BY + ORDER BY` 就能完成，但模型开始生成：

```json
{
  "cte": [{
    "name": "city_orders",
    "query": {
      "scan": "orders",
      "group": ["orders.city"],
      "agg": [{"fn": "count_all", "as": "order_count"}],
      "select": ["orders.city", "order_count"]
    }
  }],
  "scan": "city_orders",
  "select": ["city", "order_count"],
  "sort": [{"col": "order_count", "dir": "desc"}]
}
```

多了一层没有意义的 CTE 包装，生成的 SQL 虽然语义上等价，但更啰嗦、更难读，LLM judge 给的分也更低。

这就是经典的**过拟合问题**：你向 prompt 加了一个新能力的文档，模型就会过度使用它，因为它"看到了"这个新工具，并且倾向于在各种场景下使用这个最新学到的工具。

结果：**8.69 → 8.45，-0.24。**

这次退步让我意识到一个重要的模式：**每个新能力的文档必须配一个"何时不用"的反例。**

后来我在 CTE 的文档里加上了这样的说明：
> CTE 适用于：需要复用中间结果（同一个子查询被引用多次）；多步骤聚合（先聚合再聚合）；派生指标计算。
> CTE 不适用于：单步 GROUP BY 查询；只需要过滤 + 聚合的简单查询。

### Version I：稳定性修复（0.00）

Version I 主要是编译器层面的修复，以及处理 CTE 的边界案例。这轮 prompt 没有实质性改动。

结果：**8.45 → 8.45，0.00。**

零变化。这轮的价值在于把 Version H 引入的编译器问题修掉，为 Version J 打好地基。

但这轮有一个重要的副作用：编译失败率从 0.5% 上升到了 2.0%。原因是我加了一些 CTE 边界处理，误伤了一些原本正常工作的路径。这说明编译器也会有蝴蝶效应——不过通常比 prompt 更容易追踪和修复。

Version I 让我意识到一件事：在 LLM 工程里，"稳定性修复"这件事往往比新功能开发更重要，也更难做好。加新能力、写新 prompt 规则，这些是让人兴奋的工作；修 bug、提高编译成功率、消灭边界案例，这些是让人烦躁但不可或缺的工作。

真正好的工程，总是有大量看不见的、枯燥的维护工作。Forge 的 14 个 `_coerce` 修复，没有一个是设计上"聪明"的东西，它们只是在一次次的失败案例分析之后，耐心地把现实的偏差和理想的规范之间的距离填平。这种工作不上头条，但决定了系统在真实场景里能不能用。

### Version J：HAVING 精准化 + 人均模式（+0.20）

Version J 修复了两个具体的语义问题。

**HAVING 精准化。** 我发现模型在处理"按某个条件过滤聚合结果"时，经常搞不清楚聚合函数的作用对象。比如"找出订单数量超过 10 个的用户"，正确做法是：

```json
{
  "group": ["orders.user_id"],
  "agg": [{"fn": "count_all", "as": "order_count"}],
  "having": [{"col": "order_count", "op": "gt", "val": 10}]
}
```

但模型有时候会在 HAVING 里引用 `orders.user_id`（一个 GROUP BY 键，不是聚合值），或者引用完整的聚合表达式而不是 agg alias。我加了更精确的正向规则来处理这些情况。

**人均模式。** 这是一类常见的查询："每个用户的平均订单金额"、"每个城市的人均消费"。这类查询经常需要先聚合再除以计数，但模型生成的结果经常在 agg alias 的引用上出错。我加了一个专门的正向规则，描述人均模式的正确写法。

结果：**8.45 → 8.65，+0.20。**

### 小结：从 A 到 J 的教训

走完这段路，我总结出了几条 prompt 工程的规律：

| 规律 | 描述 |
|---|---|
| 正向 > 负向 | 说"应该做 X"比"不要做 Y"更稳定 |
| 新能力必须配反例 | 否则模型会过度使用新工具 |
| 精确 > 通用 | 针对具体模式的规则，比宽泛的原则更有效 |
| 改 prompt 要全量跑 | 一道题的改进可能破坏另外三道题 |
| 编译器 > prompt | 当错误有确定性的修复，写编译器比改 prompt 更可靠（后面详述）|

这张表背后有一个更深层的认识：**prompt 工程是一门经验学科，不是一门严格科学。** 没有人能事先告诉你"加了这条规则会让哪道题提升多少、会让哪道题退步多少"。这个知识只能通过大量的迭代和观察来积累，而且即使积累了很多经验，仍然会有反直觉的结果。

这并不是说 prompt 工程没有规律——上面总结的这些规律是真实的、可重复的。但它们是在一定条件下的规律，而不是普适的定律。当你换了模型、换了任务类型、换了输出格式要求，这些规律的有效性需要重新验证。

认识到这一点，让我在 prompt 迭代中变得更谦逊：不去预测每次改动的效果，只是做出改动，然后老实地看数据说话。"全量跑测试，看数据说话"是我在整个过程里最重要的工程习惯之一。

## 六、编译器的哲学：为什么编译器比 Prompt 更可靠

在整个 Forge 的开发过程中，最让我惊喜的发现之一是：**单次影响最大的改进，不是任何一次 prompt 迭代，而是一个编译器修复。**

Case 39 是一道测试窗口函数的题目。在修复之前，LLM judge 给它的平均分是 3.0——非常低，几乎每次运行都生成了语义错误的结果。我分析之后发现，问题不在 LLM 的理解，而在编译器里一个关于 alias 作用域的处理 bug。修复之后，Case 39 的分数跳到了 9.0。

**一个编译器 bug 修复，Case 39 从 3.0 到 9.0。**

这件事让我重新思考了"在 prompt 上花时间"和"在编译器上花时间"的投资回报比。

### Prompt 的蝴蝶效应

Prompt 的最大问题是它的全局性。当你修改 prompt 里的一段文字，这个修改会影响 LLM 在**所有**情况下的行为。你针对 Case 5 加了一条规则，它可能会让 Case 5 提升 0.5 分，但同时让 Case 12 和 Case 27 各降 0.3 分。净结果是负的，但你可能没有发现，因为你只看了你想解决的那个案例。

这就是蝴蝶效应。Prompt 修改的影响范围是全局的，边际效应难以预测，测试成本极高（每次都需要全量跑 40 道题，每题 5 次，算上 API 调用延迟需要等很长时间）。

### 编译器修复的确定性

编译器修复则完全不同。当我在 `_coerce()` 函数里加一个新的修复，它只影响满足特定条件的输入。修复 Fix 4（`count_all` 不允许有 `col` 字段）只在 LLM 生成了 `{"fn": "count_all", "col": "t.id", "as": "cnt"}` 这种格式时生效，完全不影响其他任何路径。

这是手术刀级别的精确修复。

### 14 个 `_coerce` 修复的诞生

到项目后期，`_coerce()` 函数里积累了 14 个修复。每一个都有一个故事：

**Fix 1：filter/having 为 dict 时包装为列表**
模型有时候生成 `"filter": {"or": [...]}` 而不是 `"filter": [{"or": [...]}]`。这是语义上等价的表达，但 schema 要求 filter 是数组。包装一下即可，不改变语义。

**Fix 2：between val 数组拆分**
`{"op": "between", "val": [500, 2000]}` 应该是 `{"op": "between", "lo": 500, "hi": 2000}`。模型经常用数组表达范围，这是直觉上合理的格式（但不符合 schema）。

**Fix 4：count_all + col 共存时删除 col**
这是一个有趣的模式。`count_all` 编译为 `COUNT(*)`，不需要 col 字段。但模型有时候会同时写上 col，可能是因为它把 `count_all` 和 `count` 的文档搞混了。直接删掉多余的 col。

**Fix 5：GROUP BY 非聚合列补齐**
这是跨方言兼容性的修复。当 SELECT 里有非聚合列，但 GROUP BY 里没有包含它，PostgreSQL 会报错（MySQL 旧版则静默返回不确定结果）。修复：在编译前自动把 SELECT 里的非聚合、非窗口、非 group_expr 别名字段补齐到 GROUP BY。

**Fix 6：having 中内联聚合函数替换**
模型有时生成 `{"col": "orders.total_amount", "fn": "avg", "op": "gt", "val": 800}` 这样的 HAVING 条件，把聚合函数内联在 having 里，而不是引用 agg alias。修复：在编译前把这类内联聚合替换为对应的 agg alias（先查找，没有的话添加一个临时 agg）。

**Fix 9：select 里的字符串内联别名转为 expr 对象**
模型有时在 select 里写字符串格式的 `"STRFTIME('%Y-%m', orders.created_at) as month"`，而不是 `{"expr": "STRFTIME('%Y-%m', orders.created_at)", "as": "month"}`。这个格式如果不处理，后续的 Fix 5 会把含有 " as month" 的完整字符串（包括空格）加入 GROUP BY，生成语法错误的 SQL。

**Fix 14：CTE 列引用前缀剥离**
这是一个比较微妙的修复。当主查询 scan 的是 CTE 名时，select/agg.col/group 里的 `table.col` 引用如果 `table` 不是 CTE 名，需要剥离 `table.` 前缀。原因：CTE 输出的列没有表前缀，`products.category` 这样的引用在 CTE 输出上会报 "no such column"。

每一个修复背后都有一个真实的失败案例，这种"来自实战的工程化"是这套系统的特色之一。

### Alias 作用域：SQL 最隐蔽的暗礁

在所有编译器工程问题里，最值得单独讲的是 alias 作用域问题。

SQL 标准有一条经常被忽视的规定：**在同层 SELECT 中，不允许引用同层定义的 alias。**

```sql
-- 这是错的：repeat_users 在 SELECT 执行时尚未定义
SELECT
  COUNT(*) AS total_users,
  COUNT(CASE WHEN order_count >= 2 THEN 1 END) AS repeat_users,
  repeat_users * 1.0 / total_users AS repurchase_rate  -- 错误！
FROM user_orders
```

直觉上，你可能觉得 `repurchase_rate` 应该能引用 `repeat_users`——毕竟它们在同一个 SELECT 列表里。但 SQL 的执行模型不允许这样做：SELECT 里所有的表达式是并行求值的，在任何一个 alias 定义之前。

在 Forge 里，LLM 经常生成 `{"expr": "repeat_users * 1.0 / total_users", "as": "repurchase_rate"}` 这样的 select 项，引用同层定义的 agg alias。直接编译成 SQL 会出错。

解决方案是 `_expand_aliases()` 函数：在编译之前，把 SELECT 中所有引用 agg/window alias 的表达式，替换为对应的完整表达式。

结果：

```sql
SELECT
  COUNT(*) AS total_users,
  COUNT(CASE WHEN order_count >= 2 THEN 1 END) AS repeat_users,
  COUNT(CASE WHEN order_count >= 2 THEN 1 END) * 1.0 / COUNT(*) AS repurchase_rate
FROM user_orders
```

这样的 SQL 是合法的：`repurchase_rate` 引用的是完整的表达式，而不是 alias。

`_expand_aliases()` 消灭了整类 alias 作用域错误，这类错误在没有这个机制的时候是编译失败率的重要来源之一。

### 为什么编译器修复 > Prompt 修复

总结这一章的核心洞察：

**当 LLM 的语义意图是对的，只是 DSL 格式稍有偏差，在编译器里加一个 `_coerce` 修复比改 prompt 更稳定。**

原因如下：

1. **作用范围精确**：编译器修复只影响满足特定输入条件的情况；prompt 修复影响所有情况
2. **可测试**：编译器修复可以写确定性的单元测试；prompt 的效果依赖 LLM 的随机性，难以用单元测试覆盖
3. **无蝴蝶效应**：编译器修复不影响其他路径；prompt 修改在全局范围内可能有副作用
4. **可追溯**：当一个编译器 bug 被发现，可以精确地写出一个重现它的测试用例；当 prompt 出现问题，很难精确追溯

这并不意味着 prompt 工程不重要——在 LLM 的语义层面，prompt 仍然是最主要的控制手段。但是，当一个错误有"确定性修复"的可能（即：给定特定的错误输入，总是能用确定的方法修复成正确输出），就应该用编译器修复，而不是 prompt 修复。

## 七、语义消歧：最后一公里

从 Version A 到 Version J，LLM Judge 评分从 7.63 提升到了 8.65。但在这个过程中，有一类错误一直没有被完全解决：**语义模糊**。

"语义模糊"是指自然语言里的一个词，可能对应多个不同的数学表达。典型的例子：

- "超过 5 次"：应该是 `op: "gt"` 还是 `op: "gte"`？（字面理解是 `gt`，即严格大于，但很多人说"超过 5 次"其实是"至少 5 次"）
- "最近 30 天"：包不包括今天？日期边界是什么？
- "高价订单"：多少算高？如果没有明确数字，应该用什么默认值？

这类问题用 prompt 很难处理，因为它们是语义层面的歧义，而不是格式层面的错误。再精确的 prompt 也无法消灭所有语义歧义，因为中文本身就有这种模糊性。

语义歧义的深层原因是：自然语言和形式语言（SQL）在表达精确性上存在根本性的差距。自然语言是为人类沟通设计的，它允许模糊和上下文推断；SQL 是为机器执行设计的，它要求每一个条件都是精确的、无歧义的。

在人类团队里，分析师遇到模糊的需求时，会去找业务方确认："你说的'超过 5 次'，是包括 5 次还是不包括？"这个确认过程消灭了歧义。但在 AI 场景里，如果每次遇到歧义都要打断流程去确认，用户体验会很差。

语义消歧库试图解决的，就是这个"确认过程自动化"的问题——不是通过对话打断，而是通过预先收集常见的语义歧义模式，在检测到歧义时自动注入最合理的解释。这比每次都打断用户去确认要高效得多，尽管它不如真实的人类确认那么准确。

### J+Sem：运行时语义消歧库

Version J+Sem 是在 J 的基础上加入了一个运行时语义消歧库。核心思路是：

**在 LLM 调用之前，根据用户的自然语言查询，注入针对性的澄清。**

比如，当用户说"下过超过 5 次订单的用户"，消歧库在 prompt 里注入：
> 注意：本查询中"超过 N 次"应解释为严格大于（op: "gt"），即下过 6 次及以上订单的用户。

这个澄清是在 LLM 生成 Forge JSON 之前注入的，所以它会影响 LLM 的输出。

这个设计的优雅之处在于：
- **不改动核心 prompt**：消歧库是独立的，不影响 prompt 的主体结构
- **无额外 API 调用**：澄清只是在调用 LLM 前往 prompt 里追加一段文字
- **精准注入**：只有当查询中检测到特定的语义模糊信号时，才注入对应的澄清；大多数查询根本不需要任何注入

结果：**8.65 → 8.82，+0.17，编译失败率从 0.5% 降到 0.0%。**

这是 Version A 到 J+Sem 整个旅程的终点：**7.63 → 8.82，总提升 +1.19 分。**

编译失败率的降低（从 3.8% 到 0.0%）同样重要。在真实部署场景里，编译失败意味着用户会收到一个错误，而不是一个（可能不太准确的）答案。把编译失败率降到零，意味着系统在最差情况下也能给出可用的输出。

### 语义消歧的局限性

语义消歧库是"无损提升"，但它也有局限：它只能处理已知的歧义模式。未知的语义模糊（比如一个行业特定的术语，在不同场景下有不同含义）仍然需要 Registry 语义层来处理。

语义消歧库和 Registry 语义层是互补的：
- **消歧库**处理语言层面的普遍歧义（"超过"是 gt 还是 gte？）
- **Registry 语义层**处理业务层面的定义歧义（"复购率"的分母是什么？）

## 八、Schema RAG：规模化的代价与解法

前面描述的所有机制，在测试 schema（四张表）上运行得很好。但真实的企业数据仓库有几十张、甚至几百张表。

把几百张表的 schema 全部塞进 prompt，有两个问题：
1. **Token 数量**：几百张表的 DDL 加起来可能有几万个 token，超过很多模型的 context window 限制
2. **信号噪音比**：大量无关的表会干扰 LLM 的意图识别，就像在一个几百人的会场里找一个认识的朋友，人越多越难找

解决方案是 Schema RAG（Retrieval-Augmented Generation）：先检索出与用户问题最相关的几张表，只把这几张表的 schema 注入 prompt。

RAG（检索增强生成）这个概念在 2023-2024 年因为知识库问答场景大火，但 Schema RAG 和传统 RAG 有几个重要的不同之处，让它比表面上看起来更复杂：

**传统 RAG** 检索的是"文档片段"，每个片段包含一段独立的知识，被检索到的片段之间可以是没有关系的。最终把多个片段拼在一起，LLM 从中提取答案。

**Schema RAG** 检索的是"表的描述"，但这些表之间是有关联结构的——用户的问题可能同时涉及 orders 表和 users 表，因为有 JOIN 关系。如果检索只召回了 orders 而没有召回 users，LLM 就没有办法写出正确的 JOIN 查询。

这意味着 Schema RAG 的召回率要求比传统 RAG 更高：在传统 RAG 里，召回 80% 的相关文档通常已经足够回答问题；在 Schema RAG 里，如果缺了一张关键的表，查询就会失败，100% 的相关表召回率才是真正的目标。

另一个不同是**检索的粒度**。传统 RAG 的检索粒度是"段落"或者"句子"，Schema RAG 的检索粒度是"整张表"——我们不能只把 users 表的部分列注入 prompt，要么注入整张表，要么不注入。这意味着 top_k 的选择尤为重要：top_k 太小，可能漏掉关键的表；top_k 太大，引入过多无关表的噪音。

Forge 的默认 top_k=5，是在"覆盖大多数需要 3-4 张表 JOIN 的查询"和"不引入太多噪音"之间的平衡。这个默认值可以根据实际数据仓库的规模和查询特点调整。

### 两级检索：向量 + BM25

Forge 实现了一套两级检索方案（`forge/retriever.py`）：

**第一级：向量检索（主要方案）。**
把每张表的描述文本（表名 + 描述 + 列名 + 枚举值）用 embedding 模型向量化，建索引。查询时，把用户问题也向量化，计算 cosine similarity，返回 top-k 个最相关的表。

**第二级：BM25-lite 关键词检索（降级方案）。**
当 embedding API 不可用时，自动切换到 BM25 关键词匹配。BM25 的中文分词策略是：纯中文块同时生成原字符串（精确匹配权重高）和字符级 bigram（模糊匹配），让"品类"能从"商品品类分类"里命中。

两级检索的触发逻辑是自动的，用户不需要任何配置：有 embedding API 就用向量检索，没有就用 BM25 降级，召回率的差距（100% vs 92.9%）在 4 张表 top_k=5 的场景下其实是零（因为所有表都在 top_k 里），只在真正的大 schema 场景下才有实质影响。

### 表描述的构建：检索质量的上限

向量检索的质量有一个硬上限：**你的表描述文本有多好，检索就有多好。**

Forge 把每张表的描述构建为：

```
Table: orders. Description: 订单主表. Columns: id, user_id, status (completed, cancelled), total_amount, created_at
```

**关键：枚举值写进描述里。** 当用户问"已完成的订单"，`status (completed, cancelled)` 里的 `completed` 让 embedding 能正确把这张表的相似度分数排高，而不是错误地命中一张叫 `finished_tasks` 的表。

这个细节是在踩了几次坑之后才意识到的。最初的版本只有表名和列名，没有枚举值，导致包含业务术语（"已完成"、"取消"等）的查询召回率明显低于预期。加入枚举值之后，召回率立刻回到接近 100%。

### Embedding API 兼容性

Forge 的 `make_embed_fn` 工厂函数屏蔽了不同 Embedding API 的格式差异。这个问题比预想的复杂，因为不同 API 厂商的请求格式差异很大：

| API | 请求格式 | 响应格式 |
|---|---|---|
| 标准 OpenAI | `{"input": ["..."]}` | `{"data": [{"embedding": [...]}]}` |
| MiniMax | `{"texts": ["..."], "type": "db"/"query"}` | `{"vectors": [[...], [...]]}` |

MiniMax 的格式差异值得特别提一下：它区分 `db`（索引文档）和 `query`（查询文本）两种嵌入类型，混用会降低召回率（因为它们是用不同的模型分支训练的，设计上是"非对称嵌入"——查询文本和文档文本在高维空间里用不同的方式表示，但仍然可以比较相似度）。Forge 自动为索引构建和查询检索使用正确的类型。

### 向量索引缓存

每次查询都调用 embedding API 会很慢（几秒钟），而且不必要——schema 不是实时变化的。

Forge 把索引缓存到 `.forge/schema_embeddings.pkl`，后续查询直接加载缓存（毫秒级）。当 Registry 里的表集合变更时（比如 `forge sync` 新增了一张表），缓存自动失效并重建。

当 `top_k >= 表总数` 时，直接返回所有表，跳过向量检索（不值得为几张表调用 API）。

### Schema 压缩效果

| 场景 | 全量 Schema Token | 检索精简后 | 减少 |
|---|---|---|---|
| 4 张表，top_k=5 | ~230 | ~230（自动全取） | 0% |
| 50 张表，top_k=5 | ~2,800 | ~560 | **~80%** |

在真实企业场景的 50 张表场景下，每次查询只注入 5 张最相关的表，schema 部分的 token 压缩了 80%。这不仅省钱，更重要的是减少了 prompt 里的噪音，LLM 可以更聚焦于真正相关的表。

## 九、Spider2：遭遇学术 Benchmark

做完了自有测试集的迭代，我开始想：如果把 Forge 放在一个外部的学术 benchmark 上，会得到什么结果？

选择 Spider2-Lite 是因为它是目前在真实企业数据仓库上最接近实战的学术 benchmark。它的查询来自真实公司的数据分析需求，而不是学术圈人工构造的例题。

我对这个测试有两种预期：

**乐观的预期**：Forge 的核心优势（消灭生成错误）在任何 schema 上都是有效的，即使是从未见过的 schema，也应该能看到相比直接 SQL 生成的提升。

**谨慎的预期**：Spider2 的查询分布可能和 Forge 的设计目标有偏差。如果 Spider2 里大量的查询都是"算法逻辑错误"类型的，Forge 的 DSL 约束不会有什么帮助。

进去跑了一轮，发现实际情况是：两个预期都部分正确，但更重要的是，评估系统本身有 bug。

### 第一轮：发现评估本身的 bug

Spider2-Lite 的 SQLite 子集有 123 个用例。我跑了第一轮，得到：
- 编译成功率：82%
- EA（Execution Accuracy）：5.9%，只有 17 道题有 gold SQL 作为参考

5.9% 是一个很低的数字，但让我怀疑的不是这个数字，而是"只有 17 道题有 gold SQL"这件事。按理说，评估标准要求所有 123 道题都应该有参考答案。

深入调查之后，发现了评估逻辑里的 bug：

**Bug 1：gold CSV 路径错误。** Spider2 的 gold 答案是 CSV 文件，每道题可能有多个子文件（`_a.csv`、`_b.csv`、`_c.csv`，对应不同的输出分片）。原来的评估代码只找精确匹配的文件名，找不到的就标记为 no_gold（没有参考答案）。实际上，很多题的 gold CSV 是多子文件格式。

**Bug 2：condition_cols 双格式解析。** Spider2 里有些题目有 `condition_cols` 字段，用来指定"哪些列的值必须完全匹配"（而不只是行数匹配）。这个字段有两种格式：per-subfile（每个子文件单独指定条件列）和 flat（所有子文件用同一个条件列列表）。原来的代码只处理了其中一种。

修复这两个 bug 之后，所有 123 道题都有了参考答案，可以进行真正的 EA 评估了。

### 完整重跑的结果

修复评估逻辑之后，完整重跑的结果：
- **编译成功率：97.6%（120/123）**
- **EA：9.2%（11/119）**
- raw SQL 兜底触发：26 次，其中 6 次通过

97.6% 的编译成功率说明 Forge DSL 的格式基本稳定，LLM 在绝大多数情况下都能生成合法的 Forge JSON。

9.2% 的 EA 则很低。我需要解释这个数字。

### 为什么 Spider2 的 EA 低？（以及为什么这不是产品缺陷）

Forge 的设计目标是解决"生成错误"和"业务逻辑错误"，不是为了解决学术 benchmark 里的算法难题。

Spider2 的查询分布和 Forge 的设计目标存在系统性错位。Spider2 里大量的查询属于我最初分类里的"算法逻辑错误"——即使有了完整的 schema，也需要了解具体的算法才能回答：

- **日期序列生成**：`WITH RECURSIVE` 或者 `generate_series`，造一张日历表，然后 LEFT JOIN 业务数据，把没有数据的日期填充为零
- **同比/环比计算**：`DATE_TRUNC` + 自关联 JOIN，或者窗口函数 LAG，比较本期和上期的数据
- **统计建模**：移动平均、线性回归的近似计算
- **复杂自关联**：同一张表和自己 JOIN 多次，比较不同时间段的同一用户行为

这些都是算法问题，不是翻译问题。Forge 的 DSL 可以表达其中一些（我后来加了窗口函数帧、递归 CTE 等能力），但即使能表达，LLM 也需要知道"解决这个问题应该用这种算法"，而这超出了 DSL 约束能覆盖的范围。

9.2% 是诚实的边界标注，不是产品缺陷。

真实企业的日常分析查询，超过 80% 属于 Forge DSL 能覆盖的范围（JOIN、过滤、聚合、窗口函数、简单的时序查询）。Spider2 代表的是分析工作的"尾部"——那些需要专业分析师花时间思考算法的复杂问题。

我在 README 里是这么写的：
> Spider2-Lite 的低 EA 是**诚实的边界标注**，不是产品缺陷。

### Spider2 带来的意外收获

虽然 EA 不高，但 Spider2 测试给 Forge 带来了一些意外的价值：

**发现了编译器的边界案例。** 三道编译失败的题（97.6% = 120/123）对应的都是 Forge DSL 目前不支持的语法，比如 LATERAL JOIN 和某些复杂的嵌套子查询。这些失败案例为 DSL 后续的能力扩展提供了方向。

**验证了 raw SQL 兜底机制的必要性。** 26 次兜底触发，说明在真实的陌生场景里，DSL 覆盖不了所有情况是完全正常的。兜底机制（当 DSL 编译失败时，让 LLM 直接生成 SQL）在实战中是必需的安全网，不应该被视为"失败"。

**泛化能力比想象中好。** Forge 的开发过程完全在四张表的 E-commerce schema 上进行，从未接触过 Spider2 的任何 schema。在完全陌生的 schema 上，97.6% 的编译成功率说明 DSL 设计本身有相当好的泛化能力——LLM 能理解 DSL 的结构并应用到新的 schema 上。

## 十、强模型的审判：GLM-5 带来的自我怀疑

Spider2 是来自学术界的挑战，而 GLM-5 测试是来自基础模型本身的挑战。

### MiniMax 测试：假说成立

在做 EA 对比测试时，我先用 MiniMax-M2.5 做了一轮：

| 方法 | EA | 正确题数 | 执行错误 |
|---|---|---|---|
| Forge（DSL）| **65.0%** | 26/40 | 2 |
| 直接 SQL | **57.5%** | 23/40 | 16 |

Forge 领先 +7.5pp，假说成立：中等能力模型在 DSL 约束下比直接生成 SQL 表现更好。

更能说明问题的是执行错误数：Forge 只有 2 个执行错误（SQL 语法正确但结果不匹配），直接 SQL 有 16 个执行错误（很多是 NOT IN + NULL 这类静默错误，以及各种语法错误）。

这正是 Forge 设计要消灭的那类错误。

### GLM-5 测试：假说被打了问号

然后我用 GLM-5（通过硅基流动 API）做了同样的测试。

遇到的第一个工程问题：`tool_choice="required"` 导致 `choices=[]`。GLM-5 的 API 在强制工具调用模式下，有时候返回空的 `choices` 数组，而不是正常的响应。原因是模型在某些情况下"拒绝了"这个工具调用，但没有提供任何替代输出。修复方法是把 `tool_choice` 从 `"required"` 改为 `"auto"`，让模型在必要时可以不使用工具（实际上它几乎总是会使用工具）。

第二个工程问题：GLM-5 极慢。这是一个推理型模型，在复杂的 CTE 题目上，单题推理时间可以高达 660 秒（接近 11 分钟）。我设置了超时，5 道题因为超时被跳过。

最终结果（35/39 题有结果）：

| 方法 | EA | 正确题数 |
|---|---|---|
| Forge（DSL）| **74.3%** | 26/35 |
| 直接 SQL | **74.4%** | 29/39 |

这个数字让我愣了很长时间。

**Forge 74.3%，直接 SQL 74.4%——几乎相同。**

更值得注意的是：由于超时，Forge 的分母是 35，直接 SQL 的分母是 39（超时分布不均，更多的超时发生在 Forge 模式的复杂 CTE 题上）。如果两组的分母相同，差距可能更小，也可能比看起来更大——样本偏差让解读变得困难。

### 不舒服的假说

这个结果指向了一个我不太想承认的假说：

**当模型足够强，"生成错误"这个错误类型本身就在萎缩。**

强推理模型（GLM-5、GPT-4o、Claude 3.5 Sonnet 这个级别）在直接生成 SQL 时，已经不经常犯 NOT IN 的 NULL 陷阱、不经常写出无类型 JOIN、不经常混淆 WHERE 和 HAVING。它们不需要 DSL 约束来避免这些错误——它自己就不会犯。

如果这个假说成立，随着基础模型持续变强，Forge 的 DSL 约束层带来的边际价值会持续缩小。

我在 README 的"正在思索中"这一节里写了这段话，没有删掉、没有粉饰，因为这是我真实的困惑。做技术研究的人，如果不敢正视数据指向的不利结论，那就很难做出真正诚实的东西。

### 什么还成立

思考了很长时间之后，我认为有三件事仍然成立，与模型能力无关：

**Registry 语义层的价值是模型无关的。** "复购率"的分母是全部用户还是有过购买记录的用户——这是业务定义问题，不是推理能力问题。再强的模型也无法凭空知道你们公司的指标定义。用了更强的 GPT-5，如果 Registry 里没有"复购率"的定义，它仍然只能猜。Registry 作为组织知识的沉淀，是真实的护城河，和模型能力强弱无关。

**审计链的价值是模型无关的。** 用户审核的 SQL 和实际执行的 SQL 是同一份。在企业数据环境里，这个"可审计、可追溯"的属性是刚需，不论 LLM 多强，监管机构和内部审计都需要能够解释"这个数字是怎么来的"。Forge 的编译链提供了这个可解释性。

**弱模型场景仍然大量存在。** 私有化部署的现实是：许多数据团队用的是本地部署的中小模型（Qwen 7B、Llama 8B），而不是 GPT-4 级别的模型。在弱模型场景下，DSL 约束的价值仍然显著——MiniMax 的测试数据（+7.5pp）就是证明。

### 还不清楚的事

- GLM-5 的"74.3% 持平"是真实信号，还是因为超时带来的样本偏差？
- 如果基础模型持续变强，Forge 的价值主张是否应该从"DSL 约束减少生成错误"转向"Registry 语义层 + 审计链"？
- DSL 是否应该变得更薄，只保留语义消歧层，放弃对 SQL 语法的约束？

这些问题目前没有答案。我把它们写在 README 里，是因为我认为一个项目的真实状态应该包括这些困惑，而不只是那些漂亮的数字。

在做 AI 产品的人里，有一种比较常见的心态：对数据的选择性解读。当测试结果支持你的假说时，大力宣传；当测试结果不支持时，找各种理由解释为什么这个数据不算数（"样本量太小"、"测试条件不完美"、"这个 benchmark 不代表真实场景"……）。这些理由有时候是真实的，但也可能是为了维护自己的先验信念而进行的合理化。

我不能保证我在分析 GLM-5 的数据时没有任何这样的倾向。但我可以做到的是：把这份不确定性公开写出来，不试图用某个"解释"来掩盖它，让读者自己判断。这是我能做到的最大程度的诚实。

这种诚实有时候很贵——它会让一些可能的合作者或者用户产生疑虑。但它比"选择性地展示有利数据"的长期代价要小得多。相信技术的人，通常也相信诚实；而那些被过度宣传吸引来的人，当他们发现真实情况时，往往会成为最激烈的批评者。

## 十一、DSL 能力扩展：把每个 SQL 设计缺陷变成机会

在解决了核心问题之后，项目进入了"填补覆盖范围"的阶段：在 Forge 的设计理念下，能不能覆盖更多的 SQL 能力？

这里有一个微妙的取舍：覆盖越多，DSL 越复杂，LLM 理解 DSL 的认知负担越大，出现新的过拟合风险越高。所以每个新能力的加入都需要认真权衡。

### 窗口函数的扩展

窗口函数是 SQL 里最难用对的功能之一，因为它的语法复杂而且跨方言不一致。最初的 Forge DSL 只支持排名类（row_number、rank、dense_rank）和聚合类（sum/avg/count over window）以及基本的导航类（lag/lead）。

第一批扩展是**窗口帧（frame）**：

```json
{
  "fn": "avg",
  "col": "orders.total_amount",
  "partition": ["orders.user_id"],
  "order": [{"col": "orders.created_at", "dir": "asc"}],
  "frame": {
    "unit": "rows",
    "start": "6 preceding",
    "end": "current_row"
  },
  "as": "rolling_7day_avg"
}
```

这段 DSL 编译为：

```sql
AVG(orders.total_amount) OVER (
  PARTITION BY orders.user_id
  ORDER BY orders.created_at
  ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
) AS rolling_7day_avg
```

7 日滚动平均是数据分析里极其常见的需求，在没有窗口帧之前，Forge 只能生成全部数据的窗口聚合（从第一行到当前行），无法做滑动窗口。

第二批扩展是**新窗口函数**：
- `percent_rank`：相对排名（0 到 1 之间），在同一组内当前行的排名百分比
- `cume_dist`：累积分布（0 到 1 之间），当前行的累积分布函数值
- `ntile(n)`：把数据分成 n 个等大的桶，返回当前行所在的桶编号（常用于分位数分析）
- `first_value` / `last_value`：返回窗口帧内第一行/最后一行的值

这些函数单独看很简单，但在数据分析里有非常清晰的用途：`ntile(4)` 可以直接做四分位数分析，`percent_rank` 可以做"当前用户消费水平在所有用户里的百分位"。

### SELECT DISTINCT 和集合运算

`distinct: true` 支持是一个一行代码的改动：

```json
{
  "scan": "orders",
  "distinct": true,
  "select": ["orders.user_id"]
}
```

编译为 `SELECT DISTINCT user_id FROM orders`。

但我在 schema.json 里加了一段警告：
> Use sparingly — DISTINCT is expensive and often indicates a missing GROUP BY.

这是基于工程经验的观察：在我见过的很多 SQL 里，`SELECT DISTINCT` 经常是错误使用的，它被用来"修补"重复行问题，而根本原因通常是 JOIN 产生了笛卡儿积，应该用 GROUP BY 而不是 DISTINCT 来去重。把这个警告写进 schema，是希望 LLM 在看到 schema 文档的时候能意识到这个陷阱。

**INTERSECT 和 EXCEPT** 集合运算：

```json
{
  "scan": "orders",
  "filter": [{"col": "orders.status", "op": "eq", "val": "completed"}],
  "select": ["orders.user_id"],
  "intersect": [{
    "query": {
      "scan": "orders",
      "filter": [{"col": "orders.created_at", "op": "gte", "val": {"$preset": "this_year"}}],
      "select": ["orders.user_id"]
    }
  }]
}
```

这段 DSL 找出"今年有已完成订单的用户"（必须同时满足两个条件）。等价于两个集合的交集。

### IN 子查询

`IN (SELECT ...)` 模式在实际查询里很常见，但在 Forge 的早期设计里是缺失的，因为它需要在 filter 条件里嵌套一个完整的子查询。

最终的设计是在 `ScalarVal`（filter 条件的值类型）里加入 `subquery` 类型：

```json
{
  "scan": "users",
  "filter": [{
    "col": "users.id",
    "op": "in",
    "val": {
      "subquery": {
        "scan": "orders",
        "filter": [{"col": "orders.total_amount", "op": "gte", "val": 1000}],
        "select": ["orders.user_id"]
      }
    }
  }],
  "select": ["users.name", "users.city"]
}
```

这段 DSL 编译为：

```sql
SELECT users.name, users.city
FROM users
WHERE users.id IN (
  SELECT orders.user_id
  FROM orders
  WHERE orders.total_amount >= 1000
)
```

值得注意的是，`anti join` 仍然比 `NOT IN + subquery` 更好——`NOT IN` 的 NULL 陷阱在子查询版本里同样存在。但 `IN + subquery` 对于"正向过滤"（找出符合某条件的记录）是合适的，NULL 陷阱只在 `NOT IN` 里才有问题。

### Agg FILTER 子句

这是 SQL 的一个相对现代的特性，SQLite 和 PostgreSQL 都支持，但 MySQL 和 BigQuery 不支持：

```json
{
  "fn": "count_all",
  "as": "vip_orders",
  "filter": [{"col": "users.is_vip", "op": "eq", "val": 1}]
}
```

编译为（SQLite/PostgreSQL）：
```sql
COUNT(*) FILTER (WHERE users.is_vip = 1) AS vip_orders
```

或者（MySQL/BigQuery 的兼容写法）：
```sql
COUNT(CASE WHEN users.is_vip = 1 THEN 1 END) AS vip_orders
```

Forge 的编译器根据方言自动选择正确的输出形式，这是方言适配矩阵的价值体现。

### 日期截断分组

日期分组是数据分析里最高频的需求之一："按月统计"、"按周统计"、"按季度统计"。不同数据库的日期截断函数不同：

| 方言 | 按月截断 |
|---|---|
| SQLite | `STRFTIME('%Y-%m', col)` |
| MySQL | `DATE_FORMAT(col, '%Y-%m')` |
| PostgreSQL | `DATE_TRUNC('month', col)` |
| BigQuery | `DATE_TRUNC(col, MONTH)` |
| Snowflake | `DATE_TRUNC('month', col)` |

Forge DSL 的 `group` 字段支持 `GroupExpr` 类型：

```json
{
  "group": [
    {"expr": "STRFTIME('%Y-%m', orders.created_at)", "as": "month"}
  ],
  "agg": [{"fn": "count_all", "as": "order_count"}],
  "select": ["month", "order_count"]
}
```

这里有一个编译器的小技巧：`month` 这个 alias 在 `group` 里定义，在 `select` 里引用。在 SQL 里，这在大多数方言里是合法的（GROUP BY alias 可以在 SELECT 里引用），但处理方式不同。Forge 的 `_expand_aliases()` 保证了这个引用能正确编译。

### 测试覆盖的扩展

随着能力的增加，单元测试也从最初的 21 个扩展到了 49 个（最新状态）。测试覆盖的原则是：每个新能力至少一个正向测试（测试正确编译），外加边界案例和跨方言测试。

49 个测试全部通过是一个重要的保障——每次改动编译器，只需要跑几秒钟的单元测试，就能确认没有引入回归。

这和 prompt 工程形成了鲜明对比：prompt 工程的"测试"需要调用 LLM API，耗时几分钟，而且结果有随机性（同样的 prompt 跑两遍可能得到不同的分数）。

## 十二、回到原点：读 Codd 和 Chamberlin

在项目进行到一半的时候，我做了一件可能是整个项目里最"无用"的事：花了一周时间读原始的数据库理论论文。

在说这段阅读经历之前，我想先说说 IRNet 这篇论文对我的启示。

### IRNet 的启示

IRNet（ACL 2019）是第一批系统性验证"中间语义表示"方法的学术工作之一。它提出了 SemQL（一种树状的语义表示语言），在 Spider benchmark 上比直接生成 SQL 的基线高了 19.5 个百分点。

读到这个结果的时候，我有一种奇特的感受：不是"啊，有人已经做过了"的沮丧，而是"我的直觉是有学术根基的"的确认。同时也让我意识到，Forge 和 IRNet 的主要区别不在于思路，而在于工程化的实现方式。

IRNet 的 SemQL 是树状的 S-expression 语法，类似这样的结构：
```
Filter(A(A(max, Product(t, sales_qty)), gt, 1000))
```

这对 2019 年的序列到序列模型来说是自然的，但在 2024 年的 LLM 时代，这种自定义语法有明显的劣势：LLM 在训练数据里几乎没见过这种格式，生成质量远不如 JSON。更重要的是，SemQL 没有任何运行时约束机制——Structured Output 在 2019 年还不存在。

IRNet 验证了方向，Forge 是用现代工具重新实现这个方向。

这个理解让我能更清晰地表述 Forge 的定位：不是重新发明轮子，而是用更现代的技术栈（JSON + Structured Output + 确定性编译器）把一个有学术根基的想法做成可以在企业里实际使用的工具。

### Codd 和关系模型的再发现

**Edgar Codd，1970年，《A Relational Model of Data for Large Shared Data Banks》。**

在读这篇论文之前，我对"关系模型"的理解是教科书层面的：表、行、列、主键、外键。读完之后我意识到，Codd 的原始论文比这深刻得多。

Codd 把数据模型的问题分成了两个层面：**物理层**（数据实际上怎么存储）和**逻辑层**（用户如何思考数据）。关系模型的核心贡献是：把这两个层面彻底分离。用户不需要知道数据是按行存储还是按列存储，不需要知道有没有索引，只需要用关系代数的语言描述"我想要什么"，系统负责"如何做到"。

这种"意图与执行的分离"，在五十年后仍然是软件工程的核心原则之一。Forge 的设计（LLM 表达意图 → 编译器确定性执行）是这个原则在 LLM 时代的一个新的应用。

有趣的是，Codd 在论文里明确指出了他的目标是解决"数据独立性"（data independence）的问题——让应用程序不依赖于数据的物理存储方式，这样数据库管理员可以优化存储结构而不破坏应用程序。五十年后，Forge 想解决的是另一种"独立性"：让查询不依赖于 LLM 的具体实现，这样换更强的 LLM 不需要改业务逻辑，而优化编译器也不需要动 LLM 的 prompt。

这两种"独立性"在本质上是相同的设计哲学：**在变化的实现和稳定的接口之间建立边界。**

这篇论文奠定了关系数据库的理论基础。在这篇论文里，Codd 提出了关系模型、原子值、第一范式……这些概念现在已经是数据领域的常识，但当时是革命性的。

Codd 原始论文里的"关系代数"（relational algebra）用的是数学符号——选择（σ）、投影（π）、连接（⋈）、差（-）、并（∪）……这和 SQL 的关键字几乎是一一对应的，只不过 SQL 把数学符号换成了更接近自然语言的单词。

**Donald Chamberlin 和 Raymond Boyce，1974年，《SEQUEL: A Structured English Query Language》。**

这篇论文描述的是 SQL 的前身，SEQUEL。有趣的是，论文里有大量篇幅解释为什么 SELECT 要写在最前面——Chamberlin 和 Boyce 做了用户实验（对，1974 年的论文里有用户实验），发现把"想要的结果"放在查询的最前面，对非程序员来说更直觉。

但这个决策创造了一个根本性的矛盾：SQL 的书写顺序（SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY）和执行顺序（FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY）是相反的。

Chamberlin 在 2012 年的一篇回顾文章里承认了这一点：
> "We made some design decisions that, in retrospect, I'm not sure were correct. The order of clauses in a SELECT statement, in particular, has confused learners for four decades."

四十年后还有人在为这个设计决策道歉。

读这些论文的收获，不是直接帮助我写出了更好的代码，而是让我理解了 SQL 的设计约束和 SQL 错误的历史来源。

**SQL 是为人类阅读而设计的**（书写顺序接近自然语言），但**LLM 生成 SQL 时面对的是语法规则**（不是阅读体验）。这两件事之间的张力，正是 Text-to-SQL 问题复杂性的根源。

Forge DSL 的字段顺序严格对齐执行顺序，是对这个历史设计决策的一次"纠正"。不是说 SQL 的设计是错的——对于人类读写来说，SELECT 在最前面确实更自然。但对于 LLM 生成的中间表示来说，执行顺序 = 字段顺序是更清晰的约束，减少了认知摩擦。

### 发现 Spider 2.0：最强模型仍然只有 21%

在阅读文献的过程中，我找到了 Spider 2.0 的论文（ICLR 2025 Oral）。Spider 2.0 是目前最接近真实企业场景的 text-to-SQL 学术 benchmark，查询来自 BigQuery 和 Snowflake 这样的真实生产数据库。

论文里有一个让我印象深刻的数字：**在 Spider 2.0 上，当前最强的基础模型，准确率只有 21%。**

21%。

这不是弱模型——这是 GPT-4o、Claude 3.5 Sonnet 这个级别的模型。在真实企业查询场景里，最强的现有模型仍然有 79% 的错误率。

这个数字一方面让我对 Forge 的方向更有信心——如果直接生成 SQL 的方法在真实场景里还有这么高的错误率，中间表示 + 约束的思路是值得追求的；另一方面也提醒了我，Forge 面对的挑战有多大。

Spider 2.0 里的那些失败案例，很大一部分属于我定义的"算法逻辑错误"——日期序列填充、复杂统计计算、多步骤的业务逻辑。这是 Forge 诚实承认的能力边界。

但剩下的那些失败，有相当一部分属于"生成错误"——LLM 知道怎么做，但 SQL 写错了。这就是 Forge 的设计空间。

### SQL 的设计缺陷目录

把 Codd 的论文、Chamberlin 的论文、以及五十年来 SQL 积累的各种边缘案例整理在一起，我总结出了一个"SQL 设计缺陷目录"，每一条对应 Forge DSL 的一个设计决策：

| SQL 设计缺陷 | 影响 | Forge 的回应 |
|---|---|---|
| 书写顺序 ≠ 执行顺序 | LLM 在推理查询逻辑时需要不断"翻转"顺序 | DSL 字段顺序 = 执行顺序 |
| NULL 三值逻辑 + NOT IN 陷阱 | 静默错误，结果集为空但不报错 | `anti` join 原语，编译为安全的 LEFT JOIN + IS NULL |
| 无类型 JOIN | 裸 JOIN = INNER JOIN，意图不明 | JOIN type 必须声明，枚举约束 |
| WHERE vs HAVING 混淆 | 聚合条件写在 WHERE 里，报错或逻辑错误 | `filter` 和 `having` 分离为独立字段 |
| GROUP BY 非聚合列不一致 | MySQL 允许但 PostgreSQL 报错 | 编译前自动补齐 GROUP BY |
| Alias 作用域 | SELECT 里的 alias 不能在同层引用 | `_expand_aliases()` 在编译前展开 |
| 方言碎片化 | 日期函数、字符串聚合、JOIN 类型跨方言不同 | 编译器根据方言输出正确代码 |
| 枚举值幻觉 | LLM 可能拼错 status 值（"complete" 而非 "completed"）| Registry 注入枚举约束，JSON Schema 强制 |

这个表格后来成了 DSL 形式化语义文档（`docs/dsl-semantics.md`）的核心内容之一。

## 十三、工程洞察：可以复用的经验

做完这个项目，有一些工程经验可以提炼出来，适用于任何 LLM + 确定性系统结合的工程项目。

### 洞察一：约束的层次

约束可以分成几个层次，每个层次的效果和成本不同：

| 约束层次 | 示例 | 约束强度 | 成本 |
|---|---|---|---|
| Token 级别约束 | JSON Schema + Structured Output | 物理不可能 | 中（需要 API 支持） |
| 编译时约束 | `_coerce()` 容错修复 | 确定性修复 | 低 |
| 运行时注入 | 语义消歧库 | 软约束（但精准） | 低 |
| Prompt 层约束 | 规则描述 | 软约束（有蝴蝶效应）| 中（需要全量测试）|

最好的策略是充分利用高层约束（token 级别），然后用编译时约束补充低频的格式偏差，最后用 prompt 层处理真正的语义问题。

Token 级别约束是最可靠的：一旦 JSON Schema 里规定 JOIN type 只能是特定枚举值，这个约束在物理上不可能被违反，不需要任何 prompt。这类约束的每一条，都节省了 prompt 里的一行规则，以及无数次的 prompt 迭代。

### 洞察二：编译器是确定性知识的容器

在 LLM + 编译器的架构里，编译器不仅仅是"把 DSL 翻译成 SQL"的简单映射。编译器是**所有确定性知识的容器**。

"NOT IN 会有 NULL 陷阱"——这不应该写在 prompt 里（因为 prompt 是软约束），而应该写在编译器里（`anti` join 固定翻译为安全的版本）。

"GROUP BY 里要补上非聚合列"——这不应该写在 prompt 里（因为不同方言的行为不同，难以在 prompt 里精确描述），而应该写在编译器里（Fix 5，在编译前自动补齐）。

"MySQL 不支持 FILTER (WHERE) 子句"——这不应该写在 prompt 里（因为 prompt 里加了这条规则，模型在 PostgreSQL 方言下也可能受影响），而应该写在编译器里（根据方言自动翻译为 CASE WHEN 等价写法）。

凡是有确定性的修复，都应该写进编译器。凡是只能用"说理"的方式影响 LLM 的，才写进 prompt。

这个原则背后有一个更深层的道理：**软件系统里的知识应该被尽可能地"硬编码"（用确定性代码表达），而不是"软配置"（用文档或 prompt 表达）。** 硬编码的知识可以测试、可以验证、可以精确追责；软配置的知识只能"希望"它起作用。

在 LLM 工程里，这个原则的应用是：把能写进代码的规则写进代码，只把真正需要 LLM 判断的事情留给 LLM。这样做的结果是：LLM 的职责变得更聚焦（只做语义理解），编译器的职责变得更清晰（只做确定性翻译），整个系统的可测试性和可维护性大幅提高。

### 洞察三：测试集的设计是战略决策

测试集设计不仅仅是"写几道测试题"，它决定了你的优化方向。

Forge 的 40 道题测试集，故意覆盖了"生成错误"最容易发生的那些类型：ANTI/SEMI JOIN、窗口函数、复杂聚合、时序查询。这意味着我们的优化是针对 Forge 设计目标的，而不是在无关的问题上浪费精力。

Spider2-Lite 的 123 道题测试集，覆盖的是学术 benchmark 里的分布，和 Forge 的设计目标有系统性错位。这不是说 Spider2 是个烂 benchmark——它是个很好的 benchmark，只是测的不是 Forge 专门设计要解决的问题。

这个区别很重要：**如果我用 Spider2 的分布设计测试集，我可能会在错误的方向上迭代，把精力花在 Forge 没法解决的算法问题上。**

### 洞察四：负向规则的危险

前面讲 prompt 迭代时已经提到了：负向规则（"不要做 Y"）比正向规则（"应该做 X"）更危险。

这背后有一个认知上的原因：负向规则告诉 LLM "哪里有危险"，但 LLM 会把"危险区域"估计得比你预想的更大，因为它想安全起见。这就像告诉一个焦虑的人"开车时不要分心"，他可能会因为担心分心而分心。

正向规则更接近于"告诉 LLM 正确的行为模式是什么"，它给 LLM 一个明确的目标，而不是一个模糊的禁区。

### 洞察五：每个新能力都有过拟合风险

这是 Version H 的教训。向 prompt 里加新能力的文档时，LLM 会有过度使用这个新能力的倾向。

这和人类学习新技术的心理类似：刚学了 CTE，就想到处用 CTE；刚学了 DISTINCT，就想到处用 DISTINCT。LLM 也有这种"新工具偏见"。

对策：每个新能力的文档里必须包含：
1. 正向示例：什么情况下用这个能力
2. 反向示例：什么情况下不要用（"尽管看起来可以用，但实际上用简单的 X 就够了"）
3. 边界说明：这个能力的局限性在哪里

### 洞察六：多轮迭代的节奏感

做了十几轮的 prompt 和编译器迭代之后，我摸索出了一套节奏：

1. **找问题**：分析失败案例，确定错误类型（生成错误 vs 语义错误 vs 算法错误）
2. **判断解法**：这个错误有确定性修复吗？有的话写编译器；没有的话考虑 prompt
3. **小改动**：每次只改一件事，控制变量
4. **全量测试**：改完立刻跑全量 40 题，看整体变化
5. **分析副作用**：有没有题目退步了？退步的原因是什么？
6. **决策**：这次改动是否值得保留？

最容易犯的错误是"大改动"——一次改了好几件事，然后发现分数变化了但不知道是哪个改动导致的。这会让调试变得极其困难。

### 洞察七：诚实的边界标注比虚假的全能更有价值

这可能是整个项目里最重要的非技术洞察。

市场上很多 AI 产品的宣传逻辑是：能力还有欠缺，但先说"我们能做到 X、Y、Z"，等产品成熟了再说局限性。这种策略在短期内看起来更吸引人，但会在用户真正使用时造成信任危机。

Forge 选择了相反的策略：从一开始就明确标注"第三类错误（算法逻辑错误）超出 Forge 能力边界"，在 README 里专门有一节解释 Spider2 的低 EA 是"诚实的边界标注"，在"正在思索中"这一节里写出对 GLM-5 数据的困惑和未解答的问题。

这种诚实有一个实际的好处：当用户理解了 Forge 的设计目标，他们就不会对边界以外的问题期望过高，也不会因为 Forge 无法解决某类问题而感到被欺骗。Forge 解决了它声称能解决的问题，这本身就是信任的来源。

## 十四、NULL 语义显式化：一个细节的哲学

在开发过程中，有一个功能是我觉得特别值得单独讲的，尽管它在整个系统里只是一个小细节：`nullable_cols` 参数。

问题的来源是这样的。假设 orders 表有一个 `discount` 列，允许 NULL（表示没有折扣）。当用户说"找出折扣金额不等于 0 的订单"时，正确的 SQL 应该是：

```sql
WHERE discount != 0 OR discount IS NULL
```

但 LLM 生成的（以及 Forge 在没有 nullable_cols 时生成的）是：

```sql
WHERE discount != 0
```

这个 SQL 会静默地过滤掉 discount 为 NULL 的行——那些根本没有折扣的订单，被当作"折扣等于 0"一样排除掉了。结果是错的，但 SQL 不报错。

`nullable_cols` 参数让调用者可以标注哪些列是可空的：

```python
compile_query(forge_json, nullable_cols=frozenset(["discount"]))
```

当 `discount` 被标注为可空，Forge 在遇到 `neq` 条件时，自动展开为：

```sql
(discount != 0 OR discount IS NULL)
```

这个功能看起来很小，但它背后有一个重要的设计哲学：

**SQL 里很多静默错误的根源是 NULL 的特殊语义没有被显式处理。** 通常，开发者知道某些列可能为 NULL，但写 SQL 的时候忘了考虑这种情况。Forge 的 `nullable_cols` 把"这个列可能是 NULL"这个业务知识显式化，让编译器在生成 SQL 时自动处理。

这和 Registry 的思路是一致的：把隐性知识（"某个列允许 NULL"）变成显性知识（在 Registry 里标注），让系统根据这个知识做出更安全的决策。

## 十五、方言适配：从三到五

Forge 最初只支持 SQLite、MySQL、PostgreSQL 三个方言。在能力扩展阶段，加入了 BigQuery 和 Snowflake。

方言适配是一件比看起来更复杂的事。不只是"日期函数不同"这么简单：

| 特性 | SQLite | MySQL | PostgreSQL | BigQuery | Snowflake |
|---|---|---|---|---|---|
| $preset 日期 | DATE('now', ...) | CURDATE() / DATE_SUB | CURRENT_DATE / INTERVAL | CURRENT_DATE() / DATE_SUB | CURRENT_DATE() / DATEADD |
| 字符串聚合 | GROUP_CONCAT | GROUP_CONCAT SEPARATOR | STRING_AGG | STRING_AGG | LISTAGG |
| FILTER (WHERE) | ✅ | ❌ | ✅ | ❌ | ❌ |
| FULL OUTER JOIN | ✅ | ❌ | ✅ | ✅ | ✅ |
| RIGHT JOIN | ✅ | ✅ | ✅ | ❌ | ✅ |
| 日期截断 | STRFTIME | DATE_FORMAT | DATE_TRUNC | DATE_TRUNC(..., MONTH) | DATE_TRUNC('month', ...) |

这个矩阵里最有趣的是 `FILTER (WHERE)` 子句：SQLite 和 PostgreSQL 原生支持，MySQL、BigQuery、Snowflake 不支持。当目标方言不支持时，Forge 编译器自动把：

```sql
COUNT(*) FILTER (WHERE is_vip = 1)
```

翻译为等价的：

```sql
COUNT(CASE WHEN is_vip = 1 THEN 1 END)
```

两种写法语义上完全等价，但对目标数据库的兼容性不同。这类"语义等价但语法不同"的翻译，正是确定性编译器的专长——LLM 不需要知道哪个方言支持什么语法，只需要表达语义意图；编译器根据目标方言自动选择正确的翻译。

RIGHT JOIN 的不支持也很有趣：BigQuery 不支持 RIGHT JOIN（可以通过调换 JOIN 顺序改成 LEFT JOIN，编译器自动处理）。这是一个我在刚开始时没想到的限制，直到在 BigQuery 方言下测试才发现。

方言适配的工程复杂度，比开发新的 DSL 能力更高。这是因为：开发新 DSL 能力是在一个可控的设计空间里工作，你知道要做什么；而方言适配是在处理历史遗留的不一致性，你不知道有多少"坑"埋在里面，只有跑测试才能发现。

这也是为什么 Forge 把方言作为 `compile_query()` 的显式参数而不是全局配置：它提醒调用者"你在为特定方言生成代码，要明确知道目标是什么"。隐式的全局配置很方便，但在多方言环境里（比如同一个团队同时使用 BigQuery 和 PostgreSQL），它会导致静默的方言不匹配错误。

## 十六、Registry 的飞轮设计

在整个 Forge 的技术栈里，Registry 是最接近"护城河"的部分。

DSL 约束、编译器、Schema RAG——这些都可以被复制（理论上）。但 Registry 里积累的组织知识，是和具体公司的数据环境深度绑定的，无法被迁移到竞争产品。

在企业软件里，"积累效应"和"切换成本"是护城河最坚实的来源之一。Registry 天然具备这两个属性：用得越多，积累的业务语义越丰富，切换到其他工具的成本就越高（因为积累的知识无法简单地迁移）。这不是靠数据绑架用户，而是靠真实的价值积累——知识多了，系统准确率确实更高，用户继续使用是因为效果好，而不是因为无法离开。

这种飞轮效应在设计上是刻意的，但它能不能真正转起来，还需要一件更基础的事情成立：**系统的基础功能要足够好，让用户愿意在上面积累知识。** 如果基础查询经常出错，用户不会有耐心去维护 Registry，飞轮就转不起来。

这也是为什么在 Forge 的开发优先级里，提高基础查询准确率始终排在"建设 Registry 语义层"之前的原因。地基不稳，上面盖什么都是危楼。

### forge sync：自动结构层

`forge sync --db sqlite:///your.db` 直连数据库，做几件事：

1. 读取所有表的 schema（列名、类型、约束）
2. 对每个低基数列（基数 < 20 的字符串/整数列）采样枚举值
3. 把结构层写入 Registry

低基数列的枚举值采样是一个关键决策。对于 `status` 列（值只有 `completed` 和 `cancelled`），把这两个值写入 Registry：

```
orders: id, user_id, status[cancelled/completed], total_amount, created_at
```

这个 `[cancelled/completed]` 注解做了几件事：
- 告诉 LLM 正确的字符串拼写（不是 `"complete"` 或 `"done"`）
- 让 Schema RAG 能在向量检索时用"已完成订单"命中 orders 表
- 提供隐式的值域约束（虽然 Structured Output 不能直接约束字符串值，但 LLM 看到了合法值的范围）

### 语义层：对话式维护

语义层是 Registry 里最有价值的部分，也是最难自动化的部分。

```
用户：复购率是指下过 2 次及以上订单的用户，除以所有下过至少 1 次订单的用户
Forge：分子 = 订单数 >= 2 的用户数，分母 = 订单数 >= 1 的用户数，是否正确？
用户：对
Forge：已保存，以后"复购率"直接可用
```

这个对话流程背后的工程实现：Forge 把"复购率"的定义结构化为：

```json
{
  "name": "repurchase_rate",
  "description": "下过2次及以上订单的用户 / 下过至少1次订单的用户",
  "numerator": {"query": "SELECT COUNT(DISTINCT user_id) FROM orders GROUP BY user_id HAVING COUNT(*) >= 2"},
  "denominator": {"query": "SELECT COUNT(DISTINCT user_id) FROM orders GROUP BY user_id HAVING COUNT(*) >= 1"}
}
```

下次用户问"复购率是多少"时，Forge 会把这个定义注入到 prompt 里，指导 LLM 生成正确的 SQL，而不是猜分母是什么。

### Registry 的正向飞轮

Registry 最重要的属性是它的**正向飞轮**效应：

- 第一次用 Forge 查复购率，Forge 可能生成错误的定义（猜分母是全部用户）
- 数据分析师发现错误，纠正（"分母是有过购买的用户"）
- 这个纠正被写入 Registry
- 下次查复购率，直接用 Registry 里的定义，不再出错
- 更多人用 Forge 查询，更多的业务知识被沉淀进 Registry
- 知识越多，错误率越低，用户越倾向于用 Forge 而不是手写 SQL

这个飞轮在大多数 AI 工具里是不存在的：通常每次对话都从零开始，之前的错误和纠正不会被记住。Forge 的 Registry 打破了这个循环。

## 十七、跨模型测试的发现与反思

做完了所有技术迭代，我想做一件事：系统性地对比 Forge 和直接 SQL 生成在不同能力模型上的表现，看看是否有规律。

这种系统性测试给了我一些意想不到的发现。

### 中等模型：差距显著

MiniMax-M2.5 是一个中等能力的模型（相当于 GPT-3.5 到 GPT-4 之间）。在 40 题测试集上：

**Forge 65.0% vs 直接 SQL 57.5%，差距 +7.5pp。**

更有启发性的是错误分布：
- Forge：2 个执行错误，0 个编译/语法错误
- 直接 SQL：16 个执行错误，1 个语法错误

直接 SQL 的 16 个执行错误里，大多数是可预测的：
- NOT IN + NULL 静默错误（4 个）
- 错误的 JOIN 类型导致多或少了记录（5 个）
- WHERE vs HAVING 混淆（3 个）
- 其他格式错误（4 个）

这些正是 Forge 设计要消灭的错误类型。数据说话，在中等模型上，Forge 的约束层确实有效。

### 强模型：差距消失

GLM-5 是一个强推理模型，在直接生成 SQL 方面的表现比 MiniMax 好得多。

**Forge 74.3% vs 直接 SQL 74.4%，几乎持平。**

当我第一次看到这个数字，我的第一反应是：数据采集是不是有问题？超时的 5 道题是不是造成了样本偏差？

仔细检查之后，确认数据是可信的。差距消失了，这是真实信号（尽管有样本量的限制）。

然后我花了很长时间思考这意味着什么。

最终我的解读是：**强模型已经内化了很多本来需要 DSL 约束的知识**。不需要有人告诉强模型"JOIN 要声明类型"，它已经知道了；不需要 schema 枚举值来防止幻觉列名，强模型在推理时会更谨慎地引用 schema 信息。

这意味着 DSL 约束的边际价值，和基础模型的能力成反比：模型越弱，约束越有价值；模型越强，约束的增量价值越小。

这是一个理性的结论，但也是一个令人不安的结论——因为基础模型在持续变强，这条曲线是单调的。

### 不变的核心

但我也找到了在这条曲线上保持不变的东西：

**Registry 语义层的价值**与模型能力无关。无论 GPT-5、GPT-6 变得多强，它们都无法知道你们公司"客单价"的定义是只算已完成订单还是所有订单。这是业务知识，不是推理能力。

**审计链的价值**与模型能力无关。在 SOX 合规环境里，数据分析师需要向审计委员会证明一个数字是怎么算出来的。"AI 生成的" 是不够的答案，需要的是"这是 SQL，这是 SQL 对应的业务逻辑，这是执行 SQL 的数据库"这样可追溯的链条。Forge 的 SQL 中间层提供了这个可追溯性。

**弱模型场景**仍然大量存在。很多数据团队用的是自部署的 Qwen 7B 或 Llama 8B，这些模型在直接生成 SQL 上的质量远不如 GPT-4 级别的模型。对于这些团队，DSL 约束的价值仍然非常显著。

### 价值主张的可能转变

基于这些反思，Forge 的价值主张可能会随着时间演化：

**现在**（弱模型和中等模型主流）：
- 主要价值：DSL 约束消灭生成错误
- 次要价值：Registry 语义层 + 审计链

**未来**（强模型主流化）：
- 主要价值：Registry 语义层（组织知识沉淀，模型无关）+ 审计链（合规需求，监管无关）
- 次要价值：在弱模型场景下的 DSL 约束

这不是价值消失，而是价值重心的转移。

## 十八、工程视角：一个诚实的中期评估

写到这里，Forge 已经有了：
- 11 个 prompt 版本（A 到 J+Sem），LLM Judge 从 7.63 提升到 8.82
- 49 个编译器单元测试，全部通过
- 14 个 `_coerce` 容错修复
- 5 个方言支持（SQLite / MySQL / PostgreSQL / BigQuery / Snowflake）
- Schema RAG（50 张表场景压缩 80% token）
- Spider2-Lite 测试（EA 9.2%，编译成功率 97.6%）
- 跨模型 EA 对比（MiniMax +7.5pp，GLM-5 ≈ 持平）

这些是真实的技术成果。但我也需要诚实地说出它现在还不是什么：

**Forge 现在还没有 Web 界面。** 所有的交互都是通过 CLI 或 Python API 完成的，这不是一个可以直接卖给业务用户的产品。对于数据工程师来说，CLI 是合理的接口；但对于数据分析师或者业务人员，他们需要一个可以直接在浏览器里使用的界面。这是 Forge 从"工程验证"变成"实际产品"的下一步关键工作。

**Forge 的 Registry 语义层还没有完整的对话式维护界面。** 目前的语义层是手动维护 JSON 文件，还没有实现"对话 → 提取结构 → 确认 → 写入"的自动流程。这意味着，虽然 Registry 的语义层在设计上很漂亮，但在实际使用里，维护它仍然需要一定的技术背景。

**Forge 还没有真正做过私有化部署的实战验证。** 所有的测试都是在本地环境或者云 API 上进行的，没有经过真实的企业内网数据库部署。私有化部署在实践中会遇到很多本地测试里看不到的问题：网络策略、权限管理、日志审计、高可用等等。这些工程问题需要在真实的企业环境里才能充分暴露和解决。

**Forge 目前的 LLM 集成方式假设了 Structured Output 的 API 支持。** 大多数主流 LLM API（OpenAI、Anthropic、国内的各主要厂商）都支持 Structured Output，但部分私有部署的本地模型（通过 vLLM 或 Ollama 运行的模型）可能不完全支持 JSON Schema 约束。对于这些场景，Forge 需要一个降级路径——用 prompt 层约束替代 token 层约束，但这会降低约束的可靠性。

这些局限不是我忘了做，而是我有意识地聚焦在"核心机制"上，先把 DSL 约束、编译器、RAG 这几个核心件做对，再考虑工程化落地。这是一个工程化项目里常见的分阶段策略：先把最关键、最难的部分做扎实，因为如果这些部分做不对，产品的外壳再漂亮也没有用。

在 AI 工具开发领域，常见的失败模式是"先做产品外壳，核心机制是黑盒"——演示效果很好，但一到真实场景就各种出问题，因为核心机制根本没有被认真验证过。我见过太多这样的工具：demo 视频里效果惊人，但真正接进公司数据库一用，错误率高得令人沮丧，而且你不知道它为什么错、也不知道怎么修复，因为里面是一个完全不透明的黑盒。

Forge 选择了相反的路：先把核心机制做扎实，可测试、可量化、有明确的边界，然后再考虑产品化。这样做的代价是进展慢，但每一个已经完成的部分都是可靠的——编译器有 49 个单元测试作为质量保证，评测指标是真实的执行准确率而不是演示截图，能力边界是诚实标注的而不是刻意模糊的。

从核心机制到完整产品，还有很长的路要走。但核心机制的正确性是一切的前提，这部分我认为是扎实的。

## 二十、一些关于 LLM 工程的泛化思考

做完 Forge 的整个开发过程，我对 LLM 工程有了一些更宏观的看法，这些看法不只适用于 Text-to-SQL，也许对其他 LLM 应用的开发者有参考价值。

### LLM 工程 ≠ Prompt 工程

在做 Forge 之前，我对 LLM 应用开发的理解主要停留在"prompt engineering"这个层面——通过精心设计 prompt，引导 LLM 产生期望的输出。这个理解没有错，但非常不完整。

真实的 LLM 工程包括：
- **约束层设计**：如何在生成阶段就把不合法的输出拦截掉（JSON Schema + Structured Output）
- **中间表示设计**：LLM 生成的输出应该是什么格式，才能既对 LLM 友好，又对下游处理友好
- **确定性处理层**：哪些逻辑应该由 LLM 处理（语义理解），哪些应该由确定性代码处理（规则转换）
- **评测体系设计**：如何衡量系统的实际效果，避免在错误的维度上优化
- **容错与降级**：当 LLM 输出不符合预期时，系统如何优雅地降级

这五件事每一件都需要大量的工程思考，都会对最终结果产生实质性影响。而 prompt 只是其中之一。

### "确定性"是工程系统的基石

在 Forge 的架构里，我刻意维护了一个明确的边界：LLM 负责非确定性的语义理解部分，编译器负责确定性的代码生成部分。

这个边界一旦模糊，整个系统就会变得难以测试和调试。比如，如果让 LLM 直接生成"更友好一点的 SQL"（而不是先生成 DSL，再由编译器生成 SQL），那么：

1. 相同的用户问题可能产生稍微不同的 SQL（因为 LLM 有随机性）
2. SQL 的正确性无法用确定性的单元测试覆盖
3. 当 SQL 出错，你不知道是 LLM 理解错了还是翻译错了

Forge 的架构把这个问题切开了：如果 Forge JSON 是对的但 SQL 是错的，那一定是编译器的 bug，写个单元测试修掉；如果 Forge JSON 是错的，那是 LLM 的问题，改 prompt 或者加语义消歧规则。两个组件的问题永远不会混在一起。

这种分离在软件工程里有个经典的名字：关注点分离（Separation of Concerns）。在 LLM 工程里，这个原则的应用是：**把 LLM 看成一个"语义提取器"，把确定性代码看成"执行引擎"，中间用一个结构化的接口连接。**

### 评测体系比想象中难

做 Forge 的过程里，我花了大量时间在评测上，而这是很多 LLM 应用开发者忽视的部分。

最初我用 LLM Judge（让另一个 LLM 给 SQL 打分）作为主要评测指标。这有几个优点：快（不需要执行 SQL）、无需 ground truth（不需要写标准答案）、覆盖范围广（LLM 能识别多种类型的 SQL 错误）。

但 LLM Judge 也有严重的局限性：
- **随机性**：同样的 SQL，Judge 给的分数每次可能不同（我见过同一道题两次运行相差 1.5 分的情况）
- **偏见**：Judge LLM 可能对某些 SQL 风格有偏好，不完全反映实际执行结果
- **不反映真正的准确率**：8.5 分的 SQL 可能在某个边界案例上仍然是错的

后来我引入了 EA（Execution Accuracy）作为更客观的指标：实际执行 SQL，与 ground truth 的结果集对比。EA 有自己的局限性（需要 ground truth，需要执行环境），但它更能反映"实际上这个 SQL 算对了没有"。

**理想的评测体系应该同时包含 LLM Judge（覆盖广、快）和 EA（客观、准确），两者互补。**

在 Spider2-Lite 测试里，EA 评估本身出现了 bug（gold CSV 路径错误、condition_cols 双格式解析问题）——这提醒了我，评测系统本身也需要被测试。一个有 bug 的评测系统，会让你在错误的优化方向上走很长时间。

### 数据管道比模型能力更重要（有时候）

这个观点可能有些反直觉。在 AI 应用开发里，大家倾向于把注意力集中在模型能力上——用更强的模型，用更好的 prompt。

但在 Forge 的开发过程里，我发现：**数据管道的质量往往比模型能力更决定结果。**

Registry 里枚举值的质量决定了幻觉列值的出现频率；Schema RAG 的召回率决定了 LLM 能不能看到正确的表；语义消歧库的覆盖范围决定了多少歧义查询能被正确处理。这些都是"数据管道"的问题，和模型能力本身关系不大。

一个很强的 LLM，如果拿到了错误的 schema（比如 `status` 列没有枚举值注解），它生成 `status = "done"` 而不是 `status = "completed"` 的概率仍然不低。一个中等的 LLM，如果拿到了完整的枚举值注解，它拼错 status 值的概率就接近零。

这不是说模型能力不重要，而是说：**在数据管道还不到位的情况下，换更强的模型带来的提升是有限的，可能远不如改进数据管道。**

### 关于"AI 原生"产品的一些想法

Forge 的设计里有一个我觉得值得单独说的模式：**让 AI 做好 AI 擅长的事，让确定性系统做好确定性系统擅长的事，不要让它们做对方的工作。**

在很多 AI 产品里，存在"把 AI 当黑盒用"的倾向——把什么问题都甩给 AI，期望 AI 自己想清楚。这在某些场景是合适的（比如创意生成），但在需要精确性和可审计性的场景里，这种做法会失控。

Forge 里的 LLM 做的是：理解自然语言意图 → 填写一个结构化的表单（Forge JSON）。这个任务 LLM 很擅长，因为它就是语言理解任务。

Forge 里的编译器做的是：把结构化的表单 → 翻译成 SQL。这个任务不需要 LLM，确定性代码更可靠，因为同样的输入永远产生同样的输出。

这种"让对的工具做对的工作"的思路，是构建可靠 AI 系统的基本原则之一。

## 二十一、那些让我意外的技术细节

开发过程中有一些技术细节，出乎意料地影响了系统的整体质量。我想把它们单独记录下来，因为这些细节往往是文档里不会写、但工程师需要踩坑才能学到的东西。

### JSON Schema 的 additionalProperties: false

这个设置听起来只是"拒绝多余的字段"，但它在实践中有更深的作用。

在开发 Forge 的早期版本里，我没有在所有地方都加 `additionalProperties: false`。结果是：LLM 开始在各种地方"发明字段"。比如：

- 在 agg 项里加 `"where"` 字段（LLM 可能觉得这是聚合的过滤条件）
- 在 window 项里加 `"filter"` 字段（想表达"只在满足某条件的行上计算窗口"）
- 在 sort 项里加 `"null_first"` 字段（想表达 NULL 值的排序位置）

这些字段有些在语义上是合理的（`null_first` 在 PostgreSQL 里确实有效），但它们不在 Forge 的 schema 里，编译器不知道该怎么处理。

加了 `additionalProperties: false` 之后，这些"发明字段"会在 JSON Schema 校验阶段被拦截，返回一个可读的错误信息（"agg 项不允许 'where' 字段，有效字段：fn, col, as, filter"），LLM 可以根据这个错误自我纠正。

这变成了一个良性循环：LLM 尝试发明字段 → schema 校验拒绝 → 错误信息告诉 LLM 正确的字段 → LLM 纠正。

### Structured Output 的 token 预算问题

Structured Output 有一个不直觉的性质：当 JSON Schema 很复杂时，schema 本身会消耗大量的 token 预算（context window 的一部分）。

Forge 的 `schema.json` 有 500+ 行，把它完整地传给 API 是有成本的。更重要的是，`schema_builder.py` 在运行时动态注入 Registry 里的枚举值（表名、列名）——如果 Registry 有 50 张表、每张表有 10 个列，这些枚举值加起来可能需要几千个 token。

解决方法：`schema_builder.py` 只注入与当前查询相关的表（通过 Schema RAG 检索），而不是所有表。这和 prompt 里的表描述压缩是同一套逻辑——只把相关的信息注入。

但这里有一个微妙的权衡：Structured Output 的枚举约束（JSON Schema 里的 `enum` 字段）是在 token 生成层面强制的，只要列名在 `enum` 里，LLM 就只能从这些值里选。如果 Schema RAG 召回错了（相关的表没有出现在 top-k 里），那个表的列名就不在枚举约束里，LLM 只能猜那张表的列名。

这意味着 **Schema RAG 的召回率是约束层有效性的上限**。100% 的召回率 → 所有相关列都在枚举约束里 → 幻觉列名物理上不可能；92.9% 的召回率（BM25-lite 降级）→ 7.1% 的概率相关表没有被召回 → 那个表的列名幻觉在少数情况下可能出现。

这是一个需要持续监控的地方。

### CTE 里的列前缀问题

这是 Fix 14 的背景，值得更详细地讲清楚。

当主查询 scan 的是一个 CTE 时，这个 CTE 的输出列是没有表前缀的。比如：

```sql
WITH user_orders AS (
  SELECT orders.user_id, COUNT(*) AS order_count
  FROM orders
  GROUP BY orders.user_id
)
SELECT orders.user_id, order_count  -- 错误！orders.user_id 不存在
FROM user_orders
```

在 CTE 的输出里，列名是 `user_id`（没有 `orders.` 前缀），不是 `orders.user_id`。但 LLM 有时候会在主查询的 select 里写 `orders.user_id`，因为它"记得"这个列来自 orders 表。

正确的写法应该是：

```sql
SELECT user_id, order_count
FROM user_orders
```

Fix 14 在编译前检查：如果主查询的 scan 是一个 CTE 名，那么 select/group/agg.col 里的 `table.col` 引用，如果 `table` 不是这个 CTE 的名字，就剥离 `table.` 前缀。

这个修复听起来简单，但它覆盖了一个非常高频的错误模式——几乎每当我引入 CTE 的测试用例，这个问题就会出现。

### 编译器里的方言条件编译

当一个功能在某些方言里不支持时，有两种处理方式：

1. **编译时报错**：检测到方言不支持这个功能，立即抛出 ValueError，带有人类可读的错误信息
2. **自动翻译**：把不支持的功能翻译成等价的方言原生写法

Forge 两种方式都用，选择哪种取决于"这个功能是否有语义等价的替代写法"：

- **FILTER (WHERE) 子句**：MySQL 不支持，但 `COUNT(*) FILTER (WHERE cond)` 和 `COUNT(CASE WHEN cond THEN 1 END)` 语义完全等价，所以自动翻译
- **FULL OUTER JOIN**：MySQL 不支持，且无法用 LEFT JOIN + UNION ALL 模拟（复杂且有风险）所以编译时报错
- **RIGHT JOIN**：BigQuery 不支持，可以通过交换左右表来改写，但这需要改变 Forge JSON 的结构，太复杂，所以编译时报错

这个决策原则是：**当语义等价翻译简单且可靠时自动翻译；当翻译复杂或有歧义时报错，让用户重新表达。**

### 窗口函数帧的字符串解析

窗口帧（Window Frame）的 `start` 和 `end` 字段是字符串，格式是：
- `"unbounded_preceding"`
- `"current_row"`
- `"6 preceding"`（N preceding）
- `"3 following"`（N following）
- `"unbounded_following"`

编译器需要把这些字符串翻译成 SQL 关键字。最初我写了一堆 `if-elif`，但这太脆弱了——任何大小写差异或者拼写变体都会导致翻译失败。

后来改成了正则匹配：

```python
import re
_FRAME_BOUND_PATTERN = re.compile(
    r'^(unbounded_preceding|current_row|unbounded_following|(\d+)\s+(preceding|following))$',
    re.IGNORECASE
)
```

这个模式覆盖了所有合法的帧边界格式，对大小写不敏感，对数字和单词之间的空格数量不敏感。

但还有一个问题：LLM 有时候生成 `"6_preceding"`（用下划线而不是空格），或者 `"6preceding"`（没有空格）。这些格式在 JSON Schema 里是合法的字符串，但正则不匹配。

解决方法是在正则匹配之前做一个简单的预处理：

```python
def _normalize_frame_bound(s: str) -> str:
    s = s.lower().strip()
    s = re.sub(r'_', ' ', s)  # 把下划线替换为空格
    s = re.sub(r'(\d+)\s*(preceding|following)', r'\1 \2', s)  # 规范化数字和单词之间的空格
    return s
```

这是"鲁棒工程"的典型模式：用宽松的输入规范化 + 严格的输出保证，而不是期望输入永远完美。

## 二十二、关于知识积累的思考

Forge 的 Registry 设计让我想到了一个更大的问题：在 AI 时代，"知识"是什么？谁拥有知识？

在前 AI 时代，数据团队的知识积累主要有三种形式：
1. **人脑里的知识**：资深分析师知道"复购率"怎么算，知道哪张表里有什么字段，这些知识在他的脑子里，他离职了就带走了
2. **文档里的知识**：数据字典、业务字典、指标手册，通常是存在 Confluence 或者某个 Wiki 里的，格式不统一，往往是过时的
3. **代码里的知识**：历史 SQL 查询、ETL 脚本，隐含了大量业务逻辑，但往往没有注释，很难被新人理解

这三种形式都有严重的问题：
- 人脑里的知识不可检索、不可复用、随人员流失而消失
- 文档里的知识通常是手动维护的，和实际的数据库状态脱节
- 代码里的知识是隐性的，需要很长时间才能被理解

Forge 的 Registry 试图解决这个问题：把知识变成**结构化的、可检索的、与数据库状态同步的**积累。

`forge sync` 解决了"结构层知识"的同步问题——数据库 schema 是真相的来源，Registry 自动与之保持一致，不需要人工维护。

语义层的对话式维护解决了"业务知识"的积累问题——每次数据分析师纠正 AI 的错误，这个纠正被结构化地写入 Registry，变成可复用的知识，而不只是一次性的对话。

这和很多 AI 工具的设计相反。大多数 AI 工具是无状态的：每次对话开始，从头开始，不记得任何历史。Forge 的 Registry 是有状态的：每次使用，知识在积累，系统在变好。

这种"正向飞轮"的设计在商业上也有意义：知识积累的越多，系统越准确，用户越倾向于用这个系统，进一步积累知识，而不是切换到竞争对手。当然，前提是这个系统真的在解决真实的问题。

### 知识的货币化

有一个关于 Registry 的问题我一直在思考但还没有答案：**谁拥有 Registry 里的知识？**

在 Forge 的设计里，Registry 是私有化部署在企业内部的，企业拥有 Registry 里的所有数据——包括 schema 结构、枚举值、业务指标定义。这和很多 SaaS AI 工具不同，那些工具的数据是存在供应商服务器上的。

私有化部署的选择是刻意的。数据团队对数据安全非常敏感——把生产数据库的 schema 发给外部 SaaS 服务，在很多公司是违反安全策略的。Forge 的私有化部署让 Registry 完全在企业自己的控制下。

但这也带来了一个有趣的问题：如果 Registry 里积累了大量高质量的业务知识（精确的指标定义、完整的枚举注解、经过验证的历史查询），这些知识的价值是非常高的。它们代表了一个数据团队几年的知识积累。这份知识如何保护、如何迁移、如何随着人员变动被传承，是 Registry 作为"组织知识库"需要回答的问题。

我没有完整的答案，但这个问题让我越来越确信：Registry 的价值不只是"让 AI 生成更准确的 SQL"，它更根本的价值是"把隐性知识变成显性资产"，让组织的数据能力不再完全依赖于少数几个掌握隐性知识的专家。

## 二十三、测试用例的哲学

在 Forge 的开发过程中，我对"如何设计测试用例"有了很多想法，这部分内容很少出现在 LLM 工程的讨论里，但它对实际结果的影响极大。

### 为什么测试集的选题比题目质量更重要

一个测试集，选题比每道题的质量更重要。选题决定了你在测量什么，以及你的优化会向哪个方向走。

Forge 的 40 道自有测试题，是我按照"生成错误"和"业务逻辑错误"的类型分类选的。每个类别有足够的题目，每道题对应一种具体的错误模式或查询类型。这意味着：

1. 当某个类别的得分下降，我能快速定位是哪种错误类型出现了问题
2. 优化是有方向的，不是在所有查询上均匀优化，而是在 Forge 设计目标覆盖的范围内优化

相比之下，Spider2-Lite 的 123 道题的分布和 Forge 的设计目标有系统性错位。如果我一开始就用 Spider2 来做优化迭代，我可能会在算法逻辑错误（Forge 的设计边界外）上花大量时间，最终发现这不是 Forge 的战场。

**测试集选题是战略决策，不是执行细节。**

### 测试用例的"死亡之吻"

在测试驱动的迭代中，有一个我称之为"死亡之吻"的现象：当你专门针对某道失败的测试题修改 prompt，你可能让这道题过了，但破坏了其他题目。

这在 Forge 的迭代里出现过很多次。Version H（新能力导致 -0.24）就是一个典型案例：我专门为 CTE 相关的题目加了 CTE 文档，结果让其他不需要 CTE 的题目也开始生成 CTE，整体分数下降。

避免"死亡之吻"的方法是：**每次修改之后必须全量跑测试，而不是只跑你修改针对的那道题。**

这听起来很基础，但在 LLM 工程里很多人做不到，因为全量测试需要调用 LLM API，有时间和金钱成本。Forge 的测试（40 题 × 5 次 = 200 次 LLM 调用）每轮需要等几分钟。这个等待时间经常让人想走捷径——只跑几道题验证一下。

我最终养成了一个习惯：无论修改多么小，一定要全量跑。有几次正是这个习惯让我发现了原本不会注意到的副作用。

### Ground Truth 的维护成本

EA（Execution Accuracy）需要 ground truth——每道题有一个标准答案 SQL，执行这个 SQL 得到结果集，然后和 Forge 生成的 SQL 的结果集对比。

维护 ground truth 有显著的成本：
- **写 ground truth SQL 本身需要时间**（而且写错了不好意识到）
- **当 schema 或数据变化时，ground truth 需要同步更新**
- **对于有多种正确写法的查询（比如 LEFT JOIN + IS NULL vs NOT EXISTS），如何判断两个结果集"等价"不是一个简单问题**

Forge 处理结果集等价的方式比较保守：对每一行每一列进行精确比较，行顺序不参与比较（因为 SQL 不保证行顺序，除非有明确的 ORDER BY）。这处理了大多数情况，但对于一些"语义等价但结果集格式不同"的情况（比如数值精度的差异）还是会有误判。

Spider2-Lite 的评测更复杂，因为它支持 `condition_cols`（只比较特定列的值），这是为了处理"结果集里有些列不用精确匹配"的情况。实现这个功能的过程中，我发现 Spider2 的 condition_cols 格式有两种（per-subfile 格式和 flat 格式），而最初的评估代码只处理了一种，导致评估逻辑出现 bug——正确答案被判为错误。

**评测系统是系统的系统，它的质量决定了你的优化方向是否正确。评测系统本身也需要充分测试。**

### 自动评测 vs 人工评测

在整个 Forge 的开发过程里，我同时使用了两种评测：
- **自动评测**：LLM Judge 打分 + EA 验证，可以快速跑大量案例
- **人工评测**：手动分析失败案例，理解失败原因

这两种评测是互补的。自动评测告诉你"整体上有没有变好"，人工评测告诉你"为什么某道题失败了，应该如何修复"。

只用自动评测的问题：你知道分数变化，但不知道原因，难以有针对性地改进。

只用人工评测的问题：费时费力，而且人类很容易有主观偏见（我刚改完 prompt，这道题肯定变好了——但其实没有）。

**最有效的工作模式**：用自动评测确认整体方向，用人工评测深入理解问题，再用自动评测验证修复效果。

## 二十四、项目的节奏与心理状态

除了技术层面，我想说一说这个项目在心理层面是什么感受。

做 Forge 不是一次线性的、一帆风顺的过程。它有很多起伏，有些起伏和技术进展关系不大，主要是心理上的。

### 早期的兴奋

最初设计 DSL 的那几天，思路特别清晰。把问题分成三类（生成错误/业务逻辑错误/算法逻辑错误），想到 JSON Schema + Structured Output 能在 token 级别施加约束，意识到书写顺序 = 执行顺序这个设计原则——这些想法来得很快，互相印证，形成了一个清晰的逻辑体系。

这种兴奋是危险的。兴奋的时候容易过度设计，把一个简单的问题弄成一个复杂的系统。我在 Version A（SQL 风格 DSL）上花了两周，结果不如直接生成 SQL——这两周的时间部分是被早期的兴奋驱动的，没有先做最小可行版本验证假说。

**教训：先做最小版本验证核心假说，再扩展。**

### 中期的平台期

从 Version G（8.69）到 Version J（8.65，中间经历了 H 的大退步和 I 的平稳）这段时间，是整个项目心理上最难熬的阶段。

分数在 8.45 附近上下浮动了很长时间，每次改动都感觉在做两步进一步退，而不是稳定地往前走。这个时候很容易陷入"我是不是在错误的方向上走"的自我怀疑。

这种情况下，有两件事帮助了我：

**第一，分析失败案例。** 不是看总体分数，而是逐题分析哪些案例失败了、为什么失败。这让我从"整体分数没变"的焦虑里抽身出来，专注于具体的、可以解决的问题。

**第二，做编译器而不是 prompt。** 在 prompt 迭代陷入困境的时候，我转而深挖编译器——分析了所有编译失败和执行失败的案例，写了几个新的 `_coerce` 修复。这些工作往往有立竿见影的效果（某道题直接从 3.0 跳到 9.0），而且不需要承担 prompt 蝴蝶效应的风险。

平台期的突破通常不是来自"更努力地做同样的事"，而是来自"换一种方式"。

### GLM-5 带来的存在主义危机

GLM-5 的测试结果（Forge 74.3% vs 直接 SQL 74.4%）是整个项目里心理冲击最大的时刻。

在这之前，我对 Forge 的核心假说（"DSL 约束减少生成错误"）是相当自信的——MiniMax 的数据（+7.5pp）清晰地支持了这个假说。GLM-5 的结果让这份自信产生了裂缝。

我花了好几天反复分析这个数据：
- 是不是超时导致的样本偏差？（5 道超时，两组分母不同）
- 是不是 GLM-5 的推理机制和 DSL 约束的交互特别不好？
- 是不是 40 道题的测试集太小，统计上不显著？

每个解释都能自圆其说，但没有一个能完全排除"强模型确实不需要 DSL 约束"这个假说。

最终，我选择了把这份不确定性写进 README，而不是试图用某个解释把它盖住。这是一个艰难的决定——因为这意味着承认"我不确定我的核心设计前提是否仍然成立"，这不是一个产品宣传稿里会说的话。

但我认为，在技术写作里，诚实是最重要的东西之一。一个把自己的不确定性藏起来的技术项目，是不可信任的。

### 后期的从容

写到现在，距离 Forge 第一行代码大概过去了四个月。项目到了一个相对稳定的状态：核心机制是清晰的，边界是诚实标注的，还有很多未解答的问题，但这些问题是已知的、可以被未来的实验回答的。

这种"从容"不是那种"我已经把所有问题解决了"的自满，而是"我知道我在做什么、为什么做、边界在哪里"的清醒。

这种清醒比兴奋更有用。

## 二十五、给同行的一些建议

如果你也在做类似的 LLM + 结构化输出的工程项目，这里有几条具体的建议，来自我的踩坑经历。

**1. 把中间表示（IR）设计放在第一位。**

LLM 的输出格式决定了很多下游问题。花时间设计一个好的 IR（中间表示），比花时间优化 prompt 的回报更高。好的 IR 应该：易于 LLM 生成（LLM 熟悉的格式，比如 JSON）；易于确定性处理（结构清晰，没有歧义）；覆盖核心需求（不需要覆盖所有情况，只需要覆盖 80% 的主要场景）。

**2. 在 token 层面施加约束，而不只是 prompt 层面。**

用 JSON Schema + Structured Output 把枚举约束做在 token 生成层面，而不是写 prompt 规则然后希望 LLM 遵守。"物理上不可能"比"我写了规则告诉它不要这么做"可靠得多。

**3. 编译器 bug 修复 > prompt 修复（对于格式类问题）。**

当 LLM 的语义意图是对的，只是格式稍有偏差，写一个 `_coerce` 修复比改 prompt 更稳定。前提是你能明确描述"什么样的输入算偏差，应该如何修正"——这需要分析大量失败案例。

**4. 正向规则替代负向规则。**

在 prompt 里描述"正确做法是什么"，而不是"不要做什么"。负向规则会让 LLM 过度保守，误伤正确行为；正向规则给 LLM 明确的目标，行为更稳定。

**5. 每个新能力必须配"何时不用"的反例。**

向 prompt 加新能力文档时，同时加上"这种情况不要用这个能力"的反例。否则 LLM 会过度使用新工具。

**6. 测试集是战略资产，不是附属品。**

精心设计的测试集，比宽泛的大量测试更有价值。每道题应该对应一种你关心的错误类型或查询模式。全量测试是习惯，不是额外的工作。

**7. 诚实地承认能力边界。**

不要假装你的系统能解决所有问题。清楚地标注出"这类问题超出系统设计边界"，不仅对用户诚实，也帮助你自己专注在真正能做好的事情上。

**8. 评测系统本身也需要测试。**

Spider2-Lite 的评估逻辑里有 bug——gold CSV 路径错误和 condition_cols 格式解析问题——导致我花了很长时间在一个不可靠的评估指标上看数字。评测系统如果有 bug，你所有的优化都可能是在朝着错误的方向走。在开始大规模评测之前，用几个你知道答案的案例验证评测逻辑的正确性。

**9. 在跨模型测试之前，先稳定单模型测试。**

我在用 MiniMax 迭代到 J+Sem 之后才引入 GLM-5 测试。如果一开始就同时在多个模型上迭代，会让问题变得复杂得多——你不知道某个 prompt 改动对哪个模型有效、对哪个模型无效，也不知道该优先为哪个模型优化。先在一个模型上建立基线，理解问题的本质，然后再做跨模型验证。

**10. 保持开发日志的习惯。**

每次迭代记录：改了什么、为什么改、结果如何、有什么副作用。这份日志在两周之后会变得非常有价值——你会记不清楚当初为什么做某个决定，这份日志是你和未来的自己之间的沟通。Forge 的这篇开发实录，其实就是从这些迭代日志里整理出来的。

## 后记：关于为什么要写这篇文章

通常，技术项目的文档是工具性的：API 文档告诉你怎么用，架构文档告诉你怎么工作，README 告诉你为什么要用。这些文档是必要的，但它们不记录过程，不记录失败，不记录那些在最终版本里消失了的错误假设。

这篇文章是不同的东西。它试图记录的，是思考的过程——不只是"我们做了什么"，而是"我们为什么这么做，我们错过了什么，我们改变了什么想法"。

做技术的人，常常觉得只有成功的经验才值得分享。失败、自我怀疑、走弯路，这些都是"不上台面"的东西，最好在项目上线之后悄悄掩埋。这种心态让很多有价值的学习消失了——后来者无从知道前人踩过哪些坑，只能自己再踩一遍。

我写这篇文章的部分原因，就是想打破这种模式。GLM-5 的持平数据让我开始怀疑核心假设，这个怀疑没有被掩盖，而是被写进了 README，被写进了这篇文章。Version H 退步了 0.24 分，这个失败没有被粉饰，而是作为"新能力导致过拟合"的一个真实案例被记录下来。

这不是谦虚的姿态，而是工程诚信的要求。如果你读了这篇文章，从中学到了什么有用的东西，那是最好的。如果你发现了我的错误或者盲区，更是我欢迎的——这些项目正需要外部的眼睛来发现单一视角的盲区。

Forge 还在做。这篇文章也会更新。
