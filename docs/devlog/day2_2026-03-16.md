### 历史债（3 月 15 日）

**NULL 语义显式化。**

起因是在测试里发现一个安静的错误：`neq` 条件在可空列上会产生语义错误。`col != 'value'` 遇到 NULL 行，NULL 既不等于也不不等于任何值，结果是这些行被静默过滤掉了。用户预期看到"所有不等于 X 的行"，实际拿到的是"所有不等于 X 的非 NULL 行"。

之所以说"安静"，是因为 SQL 执行不报错，结果集也会返回，只是行数少了。如果数据里 NULL 占比很低，这个差异会被忽略很久。

修法是在编译器里加 `nullable_cols` 参数，如果列在其中，`neq` 自动展开：

```python
# compiler.py — _condition()
if op == "neq" and nullable_cols and _col_is_nullable(col, nullable_cols):
    return f"({col} != {_val(cond['val'], dialect)} OR {col} IS NULL)"
```

`nullable_cols` 同时支持 `"table.col"` 和 `"col"` 两种格式，通过 `_col_is_nullable()` 辅助函数处理。所有递归调用路径（CTE、UNION、子查询里的 `_compile`）统一透传这个参数。测试用例从 38 个扩展到 49 个，全绿。

这件事本身不复杂，但它让我想到一个问题：还有多少类似的"安静错误"藏在 DSL 的其他部分里，没被发现，也没有测试覆盖？

**召回策略的演进。**

第一版用 BM25-lite 关键词匹配。上线之后发现一个问题：用户问"复购率是多少"，这句话里没有任何词能直接命中 `dwd_user_order_stat` 这张表。关键词匹配的假设是"用户的问题包含表名相关词汇"，但真实查询往往是业务语言，不是技术术语。召回经常打空。

改成 embedding 向量检索之后，大幅改善，但又有了新问题：语义层里定义了复购率，定义里引用了 `orders` 表，但 `orders` 可能在相似度排名里落到第 6 位，被 top_k 截断了。召回的是"相关的表"，不是"指标定义依赖的表"，这两个集合有时候不重合。

然后加了 FK 扩展和列名匹配作为补充，再然后想到了 Phase 0：如果语义层里已经有了这个指标，直接确定性地加入它依赖的所有表，不靠相似度猜。最终 `retrieve()` 的结构是：

```python
# retriever.py — retrieve()

# Phase 0：指标名称直接匹配，确定性最高
metric_tables = self._retrieve_by_metrics(question)

# Phase 1：表级语义检索（向量 or BM25 降级）
candidates = self._retrieve_by_embedding(question, embed_fn, top_k)

# Phase 2：列名关键词补充
col_tables = self._retrieve_by_columns(question, top_k)

# 合并：metric_tables 优先级最高
merged = deduplicated(metric_tables + candidates + col_tables)

# Phase 3：FK 自动扩展
return self._expand_fks(merged)
```

Phase 0 的匹配逻辑：指标名和 label 双向匹配，也拆分 label 中的词（"客单价" → "客单"、"单价"），任一命中即加入。这个逻辑不算精妙，但因为是确定性的，结果是稳定的。

### 飞书 Bot 的几个坑（3 月 15 日）

**Bot 不回复。**

发现的方式是用户发了消息，Bot 没有任何反应。查日志，没有任何报错。我一开始以为是消息没有收到，打了一些日志进去，确认消息收到了，但处理逻辑里没有输出。再往里查，发现线程里抛了异常，被 `daemon=True` 的后台线程静默吞掉了。

修法：在线程主循环里加 `try/except`，异常时发红色错误卡片给用户并打 error 日志。

**消息回复乱序。**

用户快速发了两条消息，两个线程并发跑 LLM，第二条先完成，先回复，同时两个线程同时读写同一个 session 对象，`pending_sql` 被交叉修改。

我想了一下，加锁只解决状态安全，不解决顺序——两个线程加锁后仍然可能乱序完成。真正需要的是"同一个用户的消息串行处理"。改成每用户一个 `queue.Queue`，单个 worker 顺序取出处理，30 秒无消息退出：

```python
def _user_worker(open_id: str, q: queue.Queue) -> None:
    while True:
        try:
            text = q.get(timeout=30)
        except queue.Empty:
            with _user_q_lock:
                _user_queues.pop(open_id, None)
                _user_workers.pop(open_id, None)
            break
        try:
            _dispatch(open_id, text)
        except Exception as exc:
            _send_info_card(open_id, f"❌ 处理出错：{exc}", template="red")
        finally:
            q.task_done()
```

乱序和状态污染同时消失。

**Context window 超限。**

MiniMax 400 报错：`context window exceeds limit (2013)`。全量 Registry 序列化成 tool schema 约 1.2MB，每次 LLM 调用都塞进去，必然超。

奇怪的是系统里已经有 RAG 召回了，system prompt 里只放相关的表。查了一下，发现 `_build_tools()` 还在用完整 registry——召回过滤了 system prompt，没有过滤工具定义。

```python
# agent/llm.py — 修复后
selected_tables = retriever.retrieve(current_question, q_embed_fn, top_k=10)
filtered_registry = {"tables": {t: registry["tables"][t] for t in selected_tables}}
tools = _build_tools(filtered_registry)
```

工具 schema 从 1,249,632 字符降到约 130,000 字符，压缩 90%。

**飞书卡片渲染问题。**

表格 cell 显示成 `map[content:服装鞋帽 tag:text]`，代码块 ``` 是原样纯文本。改了几次 JSON 格式没有效果，才想到去看官方文档——`lark_md` 不支持代码块，table cell 只能是纯字符串。SQL 代码改用 Pygments 渲染成 PNG 图片上传飞书；表格 cell 改成 `str(v)`。

### SQL 查询缓存（3 月 15 日）

这是 3 月 15 日工作量最大的部分。

缓存的设计起点是一个问题：如果这个缓存里存了一条错误的 SQL，会发生什么？

SQL 的错误有很多种。语法错误会报错，容易发现。更麻烦的是逻辑上正确、执行成功、只是结果和业务预期不符的 SQL。比如"复购率"的分母写成了所有注册用户，而不是下过至少一单的用户——SQL 执行不报错，数字出来了，只是偏低，要等业务发现不对才能追溯。

如果只有 Stage 1（用户确认执行），我们缓存的是"用户愿意让 Forge 执行这条 SQL"。但这是操作确认，不是业务判断。一个人点了"确认执行"，不代表他看过结果之后觉得数字是对的。

所以加了 Stage 2：用户看到结果之后，再回复"准确"，缓存条目才升级到 `verified`，才能被模糊匹配命中：

```python
# cache.py — 状态流转
add_pending()  →  status=pending    # Stage 1：执行确认
verify()       →  status=verified   # Stage 2：结果准确确认（可被模糊匹配传播）
reject()       →  status=rejected   # Stage 2：结果不准，软删除

# 查询时
lookup_exact()   # forge_json MD5 精确匹配，pending + verified 均可返回
lookup_fuzzy()   # embedding 余弦相似度，threshold=0.92，仅返回 verified
```

`pending` 条目是局部知识（只有精确命中才触发），`verified` 条目是组织知识（可以传播给相似问题）。把一条没有经过业务验证的 SQL 推给其他用户，如果是对的，省了一次 LLM 调用；如果是错的，可能让多个人基于错误数字做决策。这个风险不对等，所以需要门控。

### CROSS JOIN 支持（3 月 16 日）

测试集里有几道"找出高于平均值的用户"的题。模型知道需要先算平均值，写出的结构也是对的——两个 CTE，一个算明细，一个算全局平均。问题出在主查询里：模型把 `avg_cte.avg_val` 直接放在了 `filter` 的 `val` 字段里，而 `avg_cte` 根本没有出现在 FROM 链上，SQLite 报"no such column: avg_cte.avg_val"。

这个错误本质上是模型不知道 SQLite 的作用域规则——`WHERE` 里只能引用已经在 FROM 链上的表。

解法是在 DSL 里加 CROSS JOIN 支持——把平均值的 CTE 作为单行表 CROSS JOIN 进来，每一行都能访问 `avg_cte.avg_val`，再用 `col2` 做列对列比较：

```python
# compiler.py — _join()
jtype = join["type"]
table = join["table"]
on    = join.get("on")   # cross join 没有 on

if jtype == "cross":
    return f"CROSS JOIN {table}", []
```

同时修改了 `schema.json`，把 `cross` 加入 join type 枚举，把 `on` 从 required 里移出（CROSS JOIN 没有 ON 子句）。`cte.md` 里加了完整的双 CTE + CROSS JOIN 示例，以及 CTE 触发词列表里补了"先算"、"再按"。

```json
{
  "cte": [
    {"name": "user_totals", "query": {"scan": "orders", "group": ["orders.user_id"],
      "agg": [{"fn": "sum", "col": "orders.total_amount", "as": "total_spent"}],
      "select": ["orders.user_id", "total_spent"]}},
    {"name": "avg_val", "query": {"scan": "user_totals",
      "agg": [{"fn": "avg", "col": "user_totals.total_spent", "as": "avg_spent"}],
      "select": ["avg_spent"]}}
  ],
  "scan": "user_totals",
  "joins": [
    {"type": "inner", "table": "users", "on": {"left": "user_totals.user_id", "right": "users.id"}},
    {"type": "cross", "table": "avg_val"}
  ],
  "filter": [{"col": "user_totals.total_spent", "op": "gt", "col2": "avg_val.avg_spent"}],
  "select": ["users.name", "user_totals.total_spent"],
  "sort": [{"col": "user_totals.total_spent", "dir": "desc"}]
}
```

### HAVING alias 展开（3 月 16 日）

C27 的 Forge JSON 写得完全正确，但结果只有 2 行，正确答案是 12 行。

把 SQL 贴到 SQLite CLI 里手动跑，发现问题不在 SQL 本身，而在 SQLite 的一个边界行为：`orders` 表里有一个原始列也叫 `total_amount`，GROUP BY 之后 HAVING 里的 `total_amount` 被 SQLite 解析成了原始列（单个订单金额 ≈ 几十到几千），而不是聚合别名（用户的历史总消费）。`HAVING total_amount > 1500` 实际上变成了"最近一笔订单 > 1500"，只有 2 个用户满足，而不是"总消费 > 1500"的 12 个用户。

修法是在编译器里做 HAVING alias 展开：如果 HAVING 的 `col` 和某个 agg 的 `as` 同名，直接替换成完整的聚合表达式，不依赖 SQLite 的 alias 解析：

```python
# compiler.py — HAVING alias 展开
_having_agg_map: dict[str, str] = {
    agg["as"]: _agg_expr(agg, dialect) for agg in q.get("agg", [])
}

def _having_condition(c: dict) -> str:
    if isinstance(c, dict) and "col" in c and c.get("col") in _having_agg_map:
        c = dict(c, col=_having_agg_map[c["col"]])
    return _condition(c, dialect, nullable_cols)
```

`HAVING total_amount > 1500` → `HAVING SUM(orders.total_amount) > 1500`。歧义消除，12 行。

更新了两个测试用例的断言：

```python
# 之前
assert "HAVING n > 5" in result
# 之后
assert "HAVING COUNT(orders.id) > 5" in result  # alias 展开，消除 SQLite 歧义
```

这个 bug 是纯 SQLite 方言问题，不是 DSL 设计缺陷。但它提醒我：DSL 编译成 SQL，然后 SQL 在具体方言上执行——中间有两个翻译层，每一层都有自己的边界行为。目前测试集用的是 SQLite，生产可能是 MySQL 或 Hive，不同方言的 alias 解析规则不完全一样，这类问题在换方言时可能会重新出现。

### Reference SQL 修正与一次回退（3 月 16 日）

这部分花的时间最多，技术含量最低，但最消耗精力。

随着越来越多地检查失败题，开始发现测试集里有些题的"正确答案 SQL"本身就有问题：

- **C29**：参考 SQL 里有 `AND o.status = 'completed'`，但题目只说"VIP 用户总消费"，没有说只看完成订单。模型的理解是对的，参考 SQL 限制了范围。
- **C28**：参考 SQL 在 SELECT 里输出了 `p.category`，但题目问的是"每个商品的总销量"，没有要求按类别分组或展示类别。
- **C38**：参考 SQL 用了宏平均（`AVG(per_user_avg)`），模型理解的是微平均（`SUM(total_amount) / SUM(order_count)`）。两种算法都合理，结果不同，题目没有说明。
- **C40**：参考 SQL 用 `u.name`，但模型一直输出 `user_id`，题目只说"列出用户"，没有指定输出姓名还是 ID。

每次发现这类问题，都需要做一个判断：是模型错了，还是参考 SQL 错了，还是题目本身有歧义？这个判断在某些题上没有唯一正确答案，让人很不安——因为改参考 SQL 这个操作本身会影响准确率数字，你不知道你是在"修正标准"还是在"迁就模型"。

改完 C29 之后，顺便把"列出用户时优先输出 `name` 而不是 `user_id`"的规则加进 prompt，想着可以统一处理这类题。加了之后 C25 开始失败——MiniMax 为了取到 `name`，给一道"统计每个用户最近一笔订单"的主查询加了一个不必要的 users 表 JOIN，生成的 SQL 从约 400 字符变成了约 600 字符，触发了模型输出截断（C25 的 3 次运行全部变成不完整 SQL）。

我盯着这个情况看了一会儿，把那条 prompt 规则直接删掉了。

这是当天第三次"改了一个地方，退了另一个地方，然后还原"。

## 往深处想

### 缺陷即机会

读 Codd 论文的时候，我一直想搞清楚：SQL 的那些"坏"特性——书写顺序与执行顺序不一致、NULL 三值逻辑、GROUP BY 跨方言行为歧义——到底是 bug，还是在当时语境里合理的取舍？

读到 Chamberlin 2012 年的回顾文章时，有一段让我停下来想了很久。他说 SEQUEL 的设计原则之一是"让非程序员能够读懂"——`SELECT … FROM … WHERE` 的顺序是英语语序，先说要什么，再说从哪取，再说条件，对人类阅读更自然。他们当时面对的挑战是说服 IBM 管理层：关系型数据库可以被非技术人员使用。可读性是第一优先级。

这在 1974 年应该是对的。问题是没有人预料到 50 年后会有 LLM 需要生成这种语言。

LLM 在生成 SQL 时，`SELECT` 是第一个 token，但它在执行顺序里其实是最后才确定的东西——投影发生在所有过滤、分组、排序之后。LLM 必须在"还不知道数据是什么形状"的时候先写下输出列。这和执行顺序是倒的。

Forge DSL 的字段顺序——`scan` → `joins` → `filter` → `group` → `having` → `select`——是按执行顺序排的，不是刻意为之，只是"这样描述查询逻辑更清晰"。读完这段历史之后，我觉得这个顺序也许不只是清晰，它可能在某种程度上减少了生成时的错误。不确定，需要数据来验证，但这个想法现在对我来说很有吸引力。

SQL 的每一个设计缺陷，也许对应着 Forge DSL 的一个设计机会。`anti` join 替代 `NOT IN`（NULL 陷阱）。`nullable_cols` 显式声明（NULL 语义歧义）。HAVING alias 展开（方言差异）。这些不是在绕开 SQL，更像是在做一次补丁——只不过补丁打在了"LLM 生成"这一层，而不是 SQL 语言本身。

这个想法现在还是直觉，不是结论。

### 分层消解

召回策略四层演进的过程里，我发现自己在反复做同一件事：把"猜"的部分往后推，把"确定"的部分往前放。

- Phase 0 是确定性的：语义层里有这个指标，就直接加入对应的表，不依赖相似度。
- Phase 1 是概率性的：不知道答案，靠 embedding 相似度估计。
- Phase 2 是规则性的：FK 约束是硬知识，JOIN 范围可以推断。
- Phase 3 是启发性的：用户提问里的词可能和某张表的列名有关联。

做完之后，我有一个还没想清楚的感觉：这四层的排序，某种程度上反映了"这类知识的确定性等级"。确定性高的先执行，把不确定性留给真正不确定的部分。如果这个判断是对的，那 Registry 积累得越多，Phase 0 能覆盖的范围越大，系统对 embedding 和 LLM 的依赖就越低。

这听起来像一个飞轮，但我还没有真正看到它跑起来，所以不太敢说肯定是。

### 门控与传播

设计缓存的时候，我问了自己一个问题：如果这个缓存里存了一条错误的 SQL，会发生什么？

这让我停了一下。

一个人点了"确认执行"，不代表他看过结果之后觉得数字是对的。所以设计了两段式确认。`pending` 条目是局部知识（只有精确命中才触发），`verified` 条目是组织知识（可以传播给相似问题）。这个区分背后的假设是：把一条没有经过业务验证的 SQL 推给其他用户，如果是对的，省了一次 LLM 调用；如果是错的，可能让多个人基于错误数字做决策。风险不对等，所以需要门控。

但我也担心 Stage 2 的反馈率会很低——用户确认执行之后，再让他们多回复一条"准确"，可能大部分人直接不理。如果反馈率太低，缓存永远停在 `pending`，模糊匹配就是空的，整个机制只是一个开销。这个问题还没有答案，得等真实使用数据才知道。

### 这个项目有点失控了

Forge 最开始是一个很简单的想法：用 JSON Schema 约束 LLM 的输出，把能用结构描述的 SQL 语义都放进去，让生成错误在物理上不可能发生。这个想法现在还是对的，我也还是相信它。

但盘点一下，`agent/prompts.py` 里的 DSL 约束规则已经有二十几条了：

```
| **高于平均 → CROSS JOIN**     | 必须建两个 CTE，用 cross join 引入标量... |
| **select 只输出题目要求的字段** | 不要额外输出参与计算的中间字段...          |
| **HAVING 用 agg alias**      | having 里的列用 agg 的 as 名...         |
| **anti join 替代 NOT IN**    | 列表来自子查询时，必须用 anti join...     |
| **col2 做列对列比较**          | filter 里比较两列用 col2，不用 val...    |
```

每一条规则的背后都是一道曾经失败的测试题。规则是对的，但整体加在一起，`system_prompt` 已经是一面密密麻麻的"禁止"和"必须"。我不清楚把这些全部塞给模型，它还能不能清晰地推理。有时候感觉像是在给一个已经在努力思考的人，不断贴便利贴——贴到一定程度，便利贴本身会成为干扰。

还有一个让我不安的节奏：每次修一道失败题，大概率会引发一道之前通过的题退化。这不是 bug，是正常的——prompt 是全局的，任何修改都可能影响其他题。但这意味着测试集在某种程度上变成了"平衡博弈"而不是"线性进步"。今天 C25 的回退是第三次"改了一个地方，退了另一个地方，然后还原"。

**最开始这只是一个灵感验证。** 我想知道 DSL 约束能不能让弱模型接近强模型。结论大概是"能，但有边界"。这个验证其实已经完成了——三方对比基准，数据在那里。

然后我开始追准确率，从 75% 到 90% 到 95%。每个点都需要更多的工程量，更多的规则，更多的 prompt 调整，更多的参考 SQL 修正。投入在增加，但"我在验证一个核心假设"这件事已经变成了"我在优化一个测试集上的分数"，这两件事不完全一样。

我不太确定这条路应该走到哪里。

---

### 静默失败

飞书 Bot 的那几个坑放在一起，有一个共同的地方：表面上系统是"正常的"，只是悄悄地在出错。

线程异常被吞掉——没有报错，Bot 不回复，用户以为是网络问题。消息乱序——每条消息都回复了，只是顺序不对，状态被污染，用户以为是 Forge 的逻辑有问题。Context window 超限——错误发生在 LLM 调用里，用户看到的是"生成失败"，不知道是上下文太长了。

我不太知道怎么系统性地预防这类问题。每一个修法都是在踩到坑之后才想到的，不是提前设计出来的。我有一种感觉，这类问题可能是并发和分布式系统里不可避免的——你只能在它出现之后尽快发现，而不能在写代码时就全部预判到。

不确定这个感觉是对的，还是我在给自己的不足找借口。

## 准确率结果

### Method K — 大 Schema 三方基准（LLM Judge，3 月 15 日）

40 题 × 3 次运行，8 个类别，LLM Judge 0–10 打分。

| 方法 | 总分 | 编译失败率 | 核心特征 |
|---|---|---|---|
| b_large_sem（直出 SQL + 语义消歧库） | **8.33** | 0.0% | 语义库消歧 |
| b_large（直出 SQL） | 8.25 | 0.0% | 无约束直出 |
| **K（Forge DSL）** | 8.07 | 5.0% | DSL 约束 + RAG 召回 |

类别分布：

| 类别 | K | b_large | b_large_sem |
|---|---|---|---|
| 多表JOIN+聚合 | 8.53 | 8.33 | **9.00** |
| 复杂过滤 | 8.20 | **9.00** | 8.60 |
| 分组+HAVING | 8.27 | 8.73 | **8.87** |
| 排名与TopN | 8.27 | 8.40 | **8.53** |
| 窗口聚合 | 7.87 | 8.00 | **8.20** |
| 时序导航 | 7.53 | 8.20 | **8.67** |

**直出 SQL 整体胜出**，在 14 张核心表的 Schema 规模下，模型可以直接表达任意 SQL 语义，不受 DSL 语法限制，失败路径也少。K 的绝对优势区间是 ANTI/SEMI JOIN 和综合复杂查询——前者是因为 `anti` join 原语天然规避了 `NOT IN` 的 NULL 陷阱；后者可能是因为 DSL 的结构化约束在多表多步查询时更有帮助，但样本太少，不好下结论。

## 悬而未决

有一种做法我没有尝试：不加规则，而是让模型失败的时候自己回来修。失败了给它 error message，让它 retry，看看它能不能自己找到正确路径。规则解决已知 pattern，retry 解决未知 pattern。两种机制可能是互补的，而不是替代关系。但 retry 需要更多的 LLM 调用，成本更高，在追求低成本部署的产品里这个代价需要仔细权衡。

还有一个更根本的问题：如果准确率的瓶颈已经是"题目歧义"而不是"生成错误"，那继续堆规则的意义就很小了。Forge 的核心假设是消灭"生成错误"，但如果剩下的 5% 失败都是语义歧义，那我现在做的工程工作其实已经没有靶子了。我不确定这个临界点到了没有，但我有点担心它已经到了。

还有一件事让我有点不安：在三方基准里，强模型（直出 SQL）和有 Forge 约束的弱模型在总分上只差 0.18（8.07 vs 8.25）。随着模型能力提升，"生成错误"会被模型自身解决，Forge DSL 的相对价值在递减。另一种解读是：私有指标定义、数仓特定命名规范这些知识不在训练数据里，Registry 的价值不会被模型能力替代。我倾向于第二种，但我说不清楚倾向的理由有多少是真的，有多少是因为我已经在这件事上投入了很多时间，不愿意接受第一种的可能性。这个偏差很难消除，只能等更长时间的数据来告诉我答案。

我没有答案，只是有点累了。

### 一个没用上的调研：DeepSeek JSON mode

今天看到 DeepSeek 文档里有一个 JSON Output 功能，`response_format: {"type": "json_object"}`，第一反应是"这对我们有没有用"。

查了一下，答案是：目前没用，反而是退步。

我们现在用的是 tool calling（`llm.py` 里的 `tools` + `tool_choice: "auto"`）——模型调用一个"工具"，把 Forge JSON 作为工具参数返回，schema 约束嵌在工具的 `parameters` 里。这已经是有 schema 约束的结构化输出了。

DeepSeek 的 `json_object` 模式只保证返回合法 JSON，不做任何 schema 校验——模型想输出什么字段就输出什么，只要 JSON 语法合法就行。切过去会失去枚举约束、必填约束、字段名约束，是降级。

文档里还有一句话："使用 JSON Output 时 API 有概率返回空内容"。这个风险在生产系统里是不可接受的。

真正值得等的是 `response_format: {"type": "json_schema", "json_schema": ...}`——OpenAI structured outputs 那种，可以把 `forge/schema.json` 直接塞进去，比 tool calling 更干净，也不需要把 schema 包装成"工具"这个概念。目前 DeepSeek 文档里没看到这个模式。

这个调研没有产出代码变更，但值得记一下：**tool calling 是我们目前能做到 schema 约束的最强方式，不要轻易换**。
