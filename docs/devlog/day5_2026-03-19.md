# Day 5 — 2026-03-19

> 准确率从 62.5% 涨到 72.5%。但比数字更有价值的发现是：14 个错误里有 5 个是我们自己犯的。

---

## 做了什么

这一天做了两件大事：一是失败分析，二是系统级优化。分析先行，优化后上。

听起来很合理。但真正开始的契机是一个不舒服的问题——四个模型在 large 数据集上的准确率都不高（52.5% ~ 65.0%），而且失败的 case 有大量重合。当多个模型在同一道题上犯同一种错，问题通常不在模型。

---

## 第一步：先看我们是不是做错了

这是一个重要的态度转变。之前几天的做法是：模型错了 → 加规则 → 改 prompt → 补示例。这次倒过来：所有模型都错的题 → 先检查我们的测试设计。

结果发现了五处自己的错误。

### 1. field_conventions 表名写的是小 schema

`field_conventions.registry.yaml` 是约定库，告诉模型什么时候该过滤 `order_status='已完成'`。但里面写的表名是 `orders.status`——这是小 schema（4 张表）的名字。大 schema（200 张表）里这个字段叫 `dwd_order_detail.order_status`。

约定文件白写了。模型看到 `orders.status` 这个约定，对着 `dwd_order_detail` 的 schema 完全不知道要映射。

```yaml
# 修复前 — 小 schema 表名，在大 schema 里无人识别
applies_to: [orders.status, order_items.order_id]

# 修复后 — 大 schema 表名
applies_to: [dwd_order_detail.order_status, dwd_order_item_detail.order_id]
```

直接后果：C19（各年龄段消费排名），DeepSeek 正确过滤了已完成订单，MiniMax 两代都没过滤。差 5 倍——22242 vs 112121。

### 2. 参考 SQL 自相矛盾

Prompt 规则说："比率/占比用 `ROUND(expr, 4)` → 产出 `0.0588`"。参考 SQL 写的是 `ROUND(x * 100, 2)` → 产出 `5.88`。模型按 prompt 规则做，我们按参考 SQL 判，差 100 倍，怎么都过不了。

模型没有错，是我们内部不一致。

### 3. 退款率指标从未定义

语义层有"退货率"（退货单/完成订单），没有"退款率"（退款订单/总订单）。两个概念不同，两张表不同，连字段名都不同。模型看不到定义，只能猜——猜出了不存在的 `dwd_order_detail.refund_id`。

幻觉的根源不是模型蠢，是我们没给它信息。

### 4. 问题文字与参考 SQL 不一致

C40："各年龄段退款率（退款订单数/**总完成订单数**）"。但参考 SQL 的分母是全部订单，不过滤已完成。C26：问"最近一笔订单与上一笔的对比"，参考 SQL 返回所有 405 笔订单的 LAG。

模型按题意做，反而错了。

### 5. benchmark 和 agent 的行为不一致

Agent 生产环境有 `MAX_RETRIES=2`——编译失败后自动把错误反馈给模型，模型修正后重试，最多 3 次机会。Benchmark 是单次即死——生成一次，编译失败直接判零。

我们在用一把尺子量另一个东西。Benchmark 测的是"单次生成准确率"，但用户体验的是"产品交付准确率"。这两个指标差了 5-7 个百分点。

## 第二步：结构性归因

修完自己的错之后，M2.7 从 62.5% 涨到 65.0%。+2.5pp。

还有 14 个 case 在失败。逐个打补丁是本能反应，但这次克制住了。把 14 个失败画到一张表上：

| 根因类型 | 案例 | 共同特征 |
|----------|------|---------|
| A. 编译失败无重试 | C34, C38 | Agent 有重试，Benchmark 没有 |
| B. 输出列不匹配 | C10, C22, C36 | 行数对但列不同——问题没说要哪些列 |
| C. 约定未被应用 | C19, C7 | 规则在 prompt 里，模型没遵守 |
| D. 模式示例缺失 | C26, C28 | 不知道怎么用 DSL 表达这个模式 |
| E. 模型推理极限 | C5, C37, C39 | Forge 边界外 |

5 类问题，不是 14 个问题。

C 类最值得思考。规则明明白白写在 system prompt 里，14K 字符，模型就是没遵守。这说明一件事：**靠 prompt 措辞来保证约定被执行，这个方向是错的。** 如果约定重要到必须遵守，就应该用程序验证，不应该靠模型的"自觉"。

## 第三步：三层系统优化

### 层 1：编译重试——对齐测量

`runner.py` 加了 `--retry N` 参数。编译或 JSON 解析失败时，把错误信息拼回 messages，让模型自我修正，最多重试 N 次。和 agent.py 的逻辑完全一致。

```bash
# 无重试（旧）
python runner.py --method r --runs 3

# 有重试（新）— 对齐生产环境行为
python runner.py --method r --runs 3 --retry 2
```

效果立竿见影：编译失败率从 5% 降到 0%。120 次 API 调用全部编译成功。C34（ANTI JOIN 缺 `scan` 字段）在第 2 次尝试后通过了。EA 直接跳到 72.5%。

这不是"作弊"——产品里本来就有这个能力，只是 benchmark 没用。

### 层 2：约定 lint——程序化验证

新建了 `forge/lint.py`，三条规则：

1. 用户行为查询（消费排名、复购、消费轨迹）没过滤 `order_status='已完成'` → 警告
2. SELECT 含 `category_id` 但缺 `category_name` → 警告
3. "客单价在X到Y之间"用了 `AVG` 而不是 `WHERE` → 警告

```python
def lint_conventions(forge_json: dict, question: str) -> list[str]:
    """检查 Forge JSON 是否违反约定。返回空 = 通过。"""
    warnings = []
    _check_order_status(forge_json, question, warnings)
    _check_category_name(forge_json, warnings)
    _check_unit_price(forge_json, question, warnings)
    return warnings
```

集成到 retry 循环：JSON 解析成功后、编译前，跑一次 lint。有 warnings 就消耗一次重试机会让模型修正。

实测：lint 正确检测了 C19（消费排名缺 order_status 过滤）和 C22（category_id 代替 category_name）。但模型在修复时经常引入新错误——加了过滤条件的同时改了列名，导致编译报错。

这暴露了一个更深的问题：**检测容易，让模型修好很难。** Lint 做对了它该做的事，但修复能力取决于模型的多轮修正质量。这是下一步需要改进的方向。

### 层 3：LAG 示例补全——填补模式库空白

`window_lag.md` 原来只有一种 LAG 模式：按月聚合后打 LAG。缺两种：

```json
// 模式 2：LAG + julianday 时间差（SQLite 不支持日期直接相减）
{
  "window": [{
    "fn": "lag", "col": "orders.created_at",
    "partition": ["orders.user_id"],
    "order": [{"col": "orders.created_at", "dir": "asc"}],
    "as": "prev_order_dt"
  }],
  "select": [
    "orders.user_id", "orders.created_at",
    {"expr": "ROUND(julianday(orders.created_at) - julianday(prev_order_dt), 1)", "as": "days_gap"}
  ]
}
```

效果明确：C28（用户下单间隔天数）在补示例前，模型生成 `order_dt - LAG(order_dt)` 直接相减（SQLite 不支持）。补完后，3 个 run 全部使用 `julianday()`。参考 SQL 也对齐到 ROUND(x, 1) 后，C28 通过。

## 今天学到的

**先看自己错没错。** 14 个失败里有 5 个是我们的设计缺陷。表名写错了、参考 SQL 格式矛盾、指标没定义、题目和答案不一致、benchmark 和 agent 行为不对齐。这些不是模型问题，是工程问题。修完这些，等于把测量仪器先校准了。

**程序检查的覆盖率是 100%，模型注意力不是。** 15 条约定全塞进 14K 字符的 prompt 里，期望模型每条都遵守，这个假设本身就有问题。重要的约定应该用代码验证，而不是用自然语言"请求"。Lint 的方向是对的，即使当前模型的修复能力还跟不上。

**对齐测量。** Benchmark 应该反映产品的真实行为，而不是制造一个更严苛的考试。Agent 有重试，benchmark 就应该有重试。否则我们在优化的是一个不存在的场景。

**打补丁 vs 加系统层。** 同样是修 C28，可以在 prompt 里加一句"用 julianday"——这是补丁，只对这道题有效。也可以在示例库里补全时间差模式——这是系统层，所有涉及日期间隔的查询都受益。做系统层的意识需要刻意保持，因为补丁更快、更诱人。

## 还剩什么

12 个 case 还在失败，但性质已经不同了。

之前的失败大多是"我们做错了"。现在剩下的主要是模型能力边界：多维 GROUP BY 漏写维度（C5）、双 SEMI JOIN 条件拼不对（C39）、LEAD 后不过滤 NULL 尾行（C29）。这些是推理深度的问题，DSL 约束帮不上忙。

综合复杂查询从 0% 涨到 20-40%。这个类别曾经是全军覆没，现在 C37（累计消费阈值）和 C40（退款率）稳定通过了。

从 62.5% 到 72.5%，+10pp。但更准确的描述不是"涨了 10 个百分点"，而是"找到了 5 个自己的错误，加了 3 层系统防护，然后模型的真实能力浮现出来了"。
