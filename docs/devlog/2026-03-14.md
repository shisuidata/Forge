**遇到了什么问题**

NULL 语义显式化的实现方式有个取舍：`nullable_cols` 需要贯穿 `_compile()` → `_join()` → `_condition()` 整条调用链，包括递归调用（CTE、UNION、子查询里的 `_compile`）。函数签名改动多，但逻辑本身不复杂。

另一个问题：BigQuery 不支持 RIGHT JOIN。这是一个比较冷的知识点，容易被忽略。加了编译时报错，提示改用 LEFT JOIN（交换表顺序）。

---

**怎么解决的**

nullable_cols 采用 `frozenset[str] | None` 类型，支持 `table.col` 和 `col` 两种匹配格式，通过 `_col_is_nullable()` 辅助函数处理。递归路径里统一透传，保持一致。

BigQuery / Snowflake 的方言差异通过扩展现有的 `if dialect == ...` 分支处理，没有引入新的抽象层——YAGNI。

---

**今天收获了什么**

SQL 的设计缺陷不是"SQL 写得不好"，而是一套 1970 年代合理的设计决策在 50 年后的语境里显得过时。Codd 和 Chamberlin 当时解决的问题（让非程序员能查询数据库、数据独立性）和今天我们面对的问题（让 LLM 可靠地生成 SQL）是不同维度的问题。

更重要的一点：**SQL 的每一个设计缺陷，都是 Forge DSL 的一个设计机会。** 书写顺序 ≠ 执行顺序 → Forge DSL 的字段顺序对齐执行顺序；NULL 三值逻辑陷阱 → `anti` join 原语 + nullable_cols；窗口函数语法复杂 → 结构化 `window` 节点。

这个视角让我对 Forge 的定位更清晰了：Forge 不只是"让 LLM 不容易出错的 SQL 生成工具"，它是**对 SQL 设计缺陷的系统性修复层**，只不过修复发生在"LLM 生成"这个阶段，而不是 SQL 语言本身。
