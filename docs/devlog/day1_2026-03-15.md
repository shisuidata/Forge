### 历史债

3 月 14 日花了大半天读论文和历史，然后做了两件事。

**NULL 语义显式化。** 起因是在测试里发现，`neq` 条件在可空列上会产生语义错误——`col != 'value'` 遇到 NULL 行，NULL 既不等于也不不等于任何值，结果是这些行被静默过滤掉了。用户预期看到"所有不等于 X 的行"，实际拿到的是"所有不等于 X 的非 NULL 行"。

修法是在编译器里加 `nullable_cols` 参数，如果列在其中，`neq` 自动展开：

```python
# compiler.py — _condition()
if op == "neq" and nullable_cols and _col_is_nullable(col, nullable_cols):
    return f"({col} != {_val(cond['val'], dialect)} OR {col} IS NULL)"
```

测试用例：

```python
# 传入 nullable_cols 时，自动展开
compile_query(
    {"scan": "orders", "filter": [{"col": "orders.user_id", "op": "neq", "val": 0}],
     "select": ["orders.id"]},
    nullable_cols=frozenset(["orders.user_id"])
)
# → SELECT orders.id FROM orders WHERE (orders.user_id != 0 OR orders.user_id IS NULL)

# 不传时，保持普通形式
compile_query(
    {"scan": "orders", "filter": [{"col": "orders.status", "op": "neq", "val": "cancelled"}],
     "select": ["orders.id"]}
)
# → SELECT orders.id FROM orders WHERE orders.status != 'cancelled'
```

`nullable_cols` 支持 `"table.col"` 和 `"col"` 两种格式，通过 `_col_is_nullable()` 辅助函数处理。递归调用路径（CTE、UNION、子查询里的 `_compile`）统一透传。测试从 38 个扩展到 49 个，全绿。

**召回策略的演进。** 第一版用 BM25-lite 关键词匹配，上线之后发现一个问题：用户问"复购率是多少"，这句话里没有任何词能直接命中 `dwd_user_order_stat` 这张表。关键词匹配的假设是"用户的问题包含表名相关词汇"，但真实查询往往是业务语言，不是技术术语。召回经常打空。

改成 embedding 向量检索之后，大幅改善，但又有了新问题：语义层里定义了复购率，定义里引用了 `orders` 表，但 `orders` 可能在相似度排名里落到第 6 位，被 top_k 截断了。召回的是"相关的表"，不是"指标定义依赖的表"，这两个集合有时候不重合。

然后加了 FK 扩展和列名匹配作为补充，再然后想到了 Phase 0：如果语义层里已经有了这个指标，直接确定性地加入它依赖的所有表，不靠相似度猜。最终 `retrieve()` 的结构是：

```python
# retriever.py — retrieve()
# Phase 0：指标名称直接匹配，精度最高
metric_tables = self._retrieve_by_metrics(question)

# Phase 1：表级语义检索（向量 or BM25 降级）
candidates = self._retrieve_by_embedding(question, embed_fn, top_k)

# Phase 2：列名关键词补充
col_tables = self._retrieve_by_columns(question, top_k)

# 合并：metric_tables 优先级最高
merged = deduplicated(metric_tables + candidates + col_tables)

# Phase 3：FK 自动扩展
return self._expand_fks(merged)
```

Phase 0 的匹配逻辑：

```python
# _retrieve_by_metrics() — 指标名和 label 双向匹配
for metric_name, metric in self._metrics_registry.items():
    label = metric.get("label", "")
    hit = any((t in q_lower or q_lower in t) for t in [metric_name, label] if len(t) >= 2)
    # 也拆分 label 中的词（"客单价" → "客单"、"单价"）
    if not hit:
        for chunk in re.split(r'[（）\s/、,，()]', label):
            if len(chunk) >= 2 and chunk in q_lower:
                hit = True; break
```

### 地面泥潭

**Bot 不回复。** 发现的方式是用户发了消息，Bot 没有任何反应。查日志，没有任何报错。我一开始以为是消息没有收到，打了一些日志进去，确认消息收到了，但处理逻辑里没有输出。再往里查，发现线程里抛了异常，被 `daemon=True` 的后台线程静默吞掉了。

修法：在线程主循环里加 `try/except`，异常时发红色错误卡片给用户并打 error 日志。

**消息回复乱序。** 用户快速发了两条消息，两个线程并发跑 LLM，第二条先完成，先回复，同时两个线程同时读写同一个 session 对象，`pending_sql` 被交叉修改。

我想了一下，加锁只解决状态安全，不解决顺序——两个线程加锁后仍然可能乱序完成。真正需要的是"同一个用户的消息串行处理"。改成每用户一个 `queue.Queue`，单个 worker 顺序取出处理，30 秒无消息退出：

```python
def _user_worker(open_id: str, q: queue.Queue) -> None:
    while True:
        try:
            text = q.get(timeout=30)
        except queue.Empty:
            # 30s 无消息，退出，下次来消息再重建
            with _user_q_lock:
                _user_queues.pop(open_id, None)
                _user_workers.pop(open_id, None)
            break
        try:
            _dispatch(open_id, text)
        except Exception as exc:
            _send_info_card(open_id, f"❌ 处理出错：{exc}", template="red")
        finally:
            q.task_done()
```

乱序和状态污染同时消失。

**Context window 超限。** MiniMax 400 报错：`context window exceeds limit (2013)`。全量 Registry 序列化成 tool schema 约 1.2MB，每次 LLM 调用都塞进去，必然超。

奇怪的是系统里已经有 RAG 召回了，system prompt 里只放相关的表。查了一下，发现 `_build_tools()` 还在用完整 registry——召回过滤了 system prompt，没有过滤工具定义。

```python
# agent/llm.py — 修复前
tools = _build_tools(registry)   # 完整 200 张表，1.2MB

# 修复后
selected_tables = retriever.retrieve(current_question, q_embed_fn, top_k=10)
filtered_registry = {"tables": {t: registry["tables"][t] for t in selected_tables}}
tools = _build_tools(filtered_registry)   # ~10 张表，130KB
```

工具 schema 从 1,249,632 字符降到约 130,000 字符，压缩 90%。

## 往深处想

### 缺陷即机会

读 Codd 论文的时候，我一直想搞清楚一件事：SQL 的那些"坏"特性——书写顺序与执行顺序不一致、NULL 三值逻辑、GROUP BY 跨方言行为歧义——到底是 bug，还是在当时语境里合理的取舍？

读到 Chamberlin 2012 年的回顾文章时，有一段让我停下来想了很久。他说 SEQUEL 的设计原则之一是"让非程序员能够读懂"——`SELECT … FROM … WHERE` 的顺序是英语语序，先说要什么，再说从哪取，再说条件，对人类阅读更自然。他们当时面对的挑战是说服 IBM 管理层：关系型数据库可以被非技术人员使用。可读性是第一优先级。

这在 1974 年应该是对的。问题是没有人预料到 50 年后会有 LLM 需要生成这种语言。

我开始想：LLM 生成 SQL 时，`SELECT` 是第一个 token，但它在执行顺序里其实是最后才确定的东西——投影发生在所有过滤、分组、排序之后。LLM 在生成 SQL 时，必须在"还不知道数据是什么形状"的时候先写下输出列。这和执行顺序是倒的。

我不知道这个观察是否能完整解释 LLM 在 text-to-SQL 上的错误，可能还有很多其他原因。但 Forge DSL 的字段顺序——`from` → `joins` → `where` → `group_by` → `having` → `select`——本来就是按执行顺序排的，不是刻意为之，只是"这样描述查询逻辑更清晰"。读完这段历史之后，我觉得这个顺序也许不只是清晰，它可能在某种程度上减少了生成时的错误。不确定，需要数据来验证。

然后我想到了另一件事：SQL 的每一个设计缺陷，也许对应着 Forge DSL 的一个设计机会。`anti` join 替代 `NOT IN`。`nullable_cols` 显式声明。`window` 节点结构化表达。这些不是在绕开 SQL，而更像是在做一次补丁——只不过补丁打在了"LLM 生成"这一层，而不是 SQL 语言本身。这个想法现在对我来说还是直觉，不是结论。

### 分层消解

召回策略四层演进的过程里，我发现自己在反复做同一件事：把"猜"的部分往后推，把"确定"的部分往前放。

Phase 0 是确定性的：语义层里有这个指标，就直接加入对应的表，不依赖相似度。
Phase 1 是概率性的：不知道答案，靠 embedding 相似度估计。
Phase 2 是规则性的：FK 约束是硬知识，JOIN 范围可以推断。
Phase 3 是启发性的：用户提问里的词可能和某张表的列名有关联。

做完之后，我有一个还没想清楚的感觉：这四层的排序，某种程度上反映了"这类知识的确定性等级"。确定性高的先执行，把不确定性留给真正不确定的部分。如果这个判断是对的，那 Registry 积累得越多，Phase 0 能覆盖的范围越大，系统对 embedding 和 LLM 的依赖就越低。这听起来像一个飞轮，但我还没有真正看到这个飞轮跑起来，所以不太敢说肯定是。

### 门控与传播

设计缓存的时候，我问了自己一个问题：如果这个缓存里存了一条错误的 SQL，会发生什么？

这让我停了一下。

SQL 的错误有很多种。语法错误会报错，容易发现。更麻烦的是逻辑上正确、执行成功、只是结果和业务预期不符的 SQL。比如"复购率"的分母写成了所有注册用户，而不是下过至少一单的用户——SQL 执行不报错，数字出来了，只是偏低，要等业务发现不对才能追溯。

如果只有 Stage 1（用户确认执行），我们缓存的是"用户愿意让 Forge 执行这条 SQL"。但这是一个操作确认，不是业务判断。一个人点了"确认执行"，不代表他看过结果之后觉得数字是对的。

所以加了 Stage 2：用户看到结果之后，再回复"准确"，缓存条目才升级到 `verified`，才能被模糊匹配命中：

```python
# cache.py — 状态流转
add_pending()  →  status=pending   # Stage 1：SQL 执行确认
verify()       →  status=verified  # Stage 2：结果准确确认（可被模糊匹配传播）
reject()       →  status=rejected  # Stage 2：结果不准，软删除

# 查询时
lookup_exact()  # forge_json MD5 精确匹配，pending + verified 均可返回
lookup_fuzzy()  # embedding 余弦相似度，threshold=0.92，仅返回 verified
```

`pending` 条目是局部知识（只有精确命中才触发），`verified` 条目是组织知识（可以传播给相似问题）。把一条没有经过业务验证的 SQL 推给其他用户，如果是对的，省了一次 LLM 调用；如果是错的，可能让多个人基于错误数字做决策。这个风险不对等，所以需要门控。

不过我也担心 Stage 2 的反馈率会很低——用户确认执行之后，再让他们多回复一条"准确"，可能大部分人直接不理。如果反馈率太低，缓存永远停在 `pending`，模糊匹配就是空的，整个机制只是一个开销。这个问题还没有答案，得等真实使用数据才知道。

### 静默失败

飞书 Bot 的那几个坑放在一起，有一个共同的地方：表面上系统是"正常的"，只是悄悄地在出错。

线程异常被吞掉——没有报错，Bot 不回复，用户以为是网络问题。消息乱序——每条消息都回复了，只是顺序不对，状态被污染，用户以为是 Forge 的逻辑有问题。Context window 超限——错误发生在 LLM 调用里，用户看到的可能是"生成失败"，不知道是上下文太长了。

我不太知道怎么系统性地预防这类问题。每一个修法都是在踩到坑之后才想到的，不是提前设计出来的。我有一种感觉，这类问题可能是并发和分布式系统里不可避免的——你只能在它出现之后尽快发现，而不能在写代码时就全部预判到。

不确定这个感觉是对的，还是我在给自己的不足找借口。

## 悬而未决

有一件事让我有点不安：在 GLM-5 的测试结果里，强模型和有 Forge 约束的弱模型在某些 benchmark 上表现接近。

一种解读是：随着模型能力提升，"生成错误"会被模型自身解决，Forge DSL 的价值在递减。

另一种解读是：benchmark 上的接近不等于业务场景里的接近——私有指标定义、数仓特定命名规范这些知识不在训练数据里，Registry 的价值不会被模型能力替代。

我倾向于第二种，但我说不清楚倾向的理由有多少是真的，有多少是因为我已经在这件事上投入了很多时间，不愿意接受第一种的可能性。这个偏差很难消除，只能等 Spider2-Lite 的测试结果出来再看。

## Method K 大 Schema 三方基准结果

今天跑完了大 Schema 三方对比基准：K（Forge DSL + 14 张电商数仓核心表）、b_large（直出 SQL，无语义库）、b_large_sem（直出 SQL + 语义消歧库）。测试集：40 题 × 3 次运行，8 个类别，LLM Judge 0–10 打分。

| 方法 | 总分 | 编译失败率 | 核心特征 |
|---|---|---|---|
| **b_large_sem** | **8.33** | 0.0% | 直出 SQL + 语义消歧库 |
| **b_large** | **8.25** | 0.0% | 直出 SQL，无语义库 |
| **K** | **8.07** | 5.0% | Forge DSL + 大 Schema |

### 分类对比

| 类别 | 用例数 | K | b_large | b_large_sem |
|---|---|---|---|---|
| 多表JOIN+聚合 | 5 | 8.53 | 8.33 | **9.00** |
| 复杂过滤 | 5 | 8.20 | **9.00** | 8.60 |
| 分组+HAVING | 5 | 8.27 | 8.73 | **8.87** |
| 排名与TopN | 5 | 8.27 | 8.40 | **8.53** |
| 窗口聚合 | 5 | 7.87 | 8.00 | **8.20** |
| 时序导航 | 5 | 7.53 | 8.20 | **8.67** |
| ANTI/SEMI JOIN | 5 | **7.40** | 7.20 | 7.13 |
| 综合复杂查询 | 5 | **8.47** | 8.14 | 7.60 |
| **总体** | **40** | **8.07** | **8.25** | **8.33** |

### 结果解读

**直出 SQL 整体胜出**——这个结论在这个特定条件下成立：14 张核心表，Schema 规模尚在模型可以"记住"的范围内。Forge DSL 的多层介入（RAG 召回 → DSL 生成 → 编译）在 Schema 选择上增加了失败路径，而直出 SQL 可以直接表达任意 SQL 语义，不受 DSL 语法限制。

**语义消歧库在大 Schema 场景的效果**：净收益 +0.08，但效果不均匀：
- 显著修复：`pay_amount` vs `total_amount` 的歧义（Cases 1,2,4,5,15）：语义库明确 GMV/客单价应用 `total_amount`，直出 SQL 原先频繁用 `pay_amount`
- 修复时序导航 Case 30：b_large Run3 生成了含空格的列别名（`ring growth`，SQL 语法错误），语义库版本消除了此类错误
- 副作用：Case 7"客单价在 500–2000 之间"被解读为用户平均客单价（GROUP BY/HAVING），改变了查询语义（9.0→7.0）；Case 36 一次 API 调用返回空响应（6.0→运气影响）

**K 的优势区间**：在需要跨 5+ 张表的综合复杂查询（8.47）和 ANTI/SEMI JOIN（7.40）上，Forge 的 `anti` join 原语仍然优于直出 SQL 的 `NOT IN`（后者有 NULL 陷阱）。两种直出 SQL 在 ANTI/SEMI JOIN 类别全部使用 `NOT IN`，K 在这个子类是唯一真正安全的方案。

两个主要失分模式（K）：

1. **字段语义歧义**：`total_amount`（订单总额）vs `actual_amount`（订单商品实际支付）——问题问"销售额"时，模型选哪个取决于它对业务的理解，不总是对的。
2. **分步计算的 alias 引用**：LAG/LEAD 引用同层 SELECT 的 alias（`LEAD(order_count) OVER ...`），SQLite 要求先 CTE 固定，模型不总是知道这个约束。
